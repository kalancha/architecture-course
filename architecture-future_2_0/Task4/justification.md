# Обоснование Terraform-конфигурации

## Почему выбраны такие ресурсы

| Ресурс | Решение | Обоснование |
|---|---|---|
| VPC Network | Отдельная сеть | Изолирует учебный контур и позволяет независимо развивать сетевую схему. |
| Subnet | `10.20.10.0/24`, одна зона | Диапазона достаточно для прототипа; одна зона снижает стоимость учебной проверки. Для production нужны несколько зон. |
| Security Group | Внутренний трафик, исходящий доступ, SSH только по явному списку | Реализует принцип наименьших привилегий. По умолчанию публичный SSH закрыт. |
| VM | `standard-v3`, 2 vCPU, 2 ГБ RAM, 20% core fraction | Минимальная конфигурация для проверки cloud-init и сетевого доступа без лишних затрат. Это не production sizing. |
| Boot Disk | 15 ГБ `network-hdd` | Достаточно для Ubuntu и служебных файлов учебной ВМ. HDD дешевле SSD и соответствует характеру нагрузки. |
| Data Disk | 20 ГБ `network-hdd` | Отделяет жизненный цикл данных от ОС. `auto_delete = false` предотвращает случайную потерю данных при пересоздании ВМ. |
| Dynamic NAT | Включается переменной | Даёт исходящий доступ к обновлениям и API без отдельного NAT-инстанса. Для production предпочтительны приватные ВМ и централизованный egress/NAT Gateway. |
| cloud-init | Пользователь и публичный SSH-ключ | Исключает ручную настройку после создания ВМ и делает начальное состояние воспроизводимым. |

## Что автоматизировано, а что остаётся ручным

Terraform управляет сетью, подсетью, группой безопасности, поиском образа, двумя дисками, ВМ, сетевым интерфейсом, NAT-признаком и cloud-init.

Вручную подготавливаются только внешние предпосылки:

1. аккаунт и биллинг;
2. отдельный учебный cloud/folder;
3. IAM-права выполняющего субъекта;
4. временный токен в переменной окружения;
5. локальная SSH-пара;
6. фактический запуск и сохранение скриншота результата.

Эти действия не кодируются в проекте, потому что затрагивают корневые полномочия, платёжный аккаунт или локальные секреты.

## Почему декларативный подход

В конфигурации зафиксировано желаемое состояние, а не последовательность команд создания. Terraform сам строит граф зависимостей по ссылкам на идентификаторы ресурсов. Это даёт:

- одинаковое развёртывание на разных итерациях;
- предварительный просмотр изменений через `terraform plan`;
- контролируемое изменение и удаление ресурсов;
- проверяемую историю в Git;
- возможность параметризовать окружения без копирования конфигурации;
- снижение количества расхождений, возникающих при ручной настройке.

## Воспроизводимость и масштабирование

Переменные отделяют топологию от параметров: можно менять зону, CIDR, размеры VM и дисков, тип окружения и правила SSH. Общие labels позволяют искать ресурсы и учитывать расходы по проекту.

Для следующего уровня зрелости конфигурацию следует разнести по модулям, хранить state в защищённом объектном хранилище с блокировками, запускать `fmt/validate/plan` в CI и применять только после ревью. Масштабирование приложения потребует instance group и балансировщика, а отказоустойчивость — размещения минимум в двух-трёх зонах.

## NAT и безопасность

`nat = true` выбран для учебной проверки, потому что ВМ требуется исходящий доступ, а отдельный NAT-инстанс непропорционально усложнил бы задание. Публичный IP сам по себе не открывает SSH: входящее правило создаётся только когда `allowed_ssh_cidrs` непустой. Нельзя указывать `0.0.0.0/0`; следует добавлять только адрес проверяющего `/32` и удалять его после проверки.

Токен не объявлен Terraform-переменной и не записывается в `terraform.tfvars`. Провайдер получает его из `YC_TOKEN`. State, планы, приватные ключи и service-account JSON исключены через `.gitignore`.
