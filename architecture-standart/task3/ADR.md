### **Решение**

#### Логика принятых решений

**1. Асинхронная интеграция ИБ → АБС через Kafka**

Команда АБС прямо указывает, что БД Oracle уже перегружена. Если подключить ИБ к АБС синхронно, онлайн-заявки при пиковых нагрузках могут привести к деградации всего банка. Поэтому для передачи заявок выбрана асинхронная модель через Kafka — она развязывает системы и гарантирует доставку заявок даже при временной недоступности АБС (R3, R4).

**2. Промежуточный Сервис заявок (Java Spring Boot)**

ИБ работает на .NET Framework 4.5, для которого нет совместимых Kafka-клиентов (+R6). Чтобы не трогать ядро ИБ у подрядчика, вводится отдельный легковесный сервис на Java Spring Boot. Этот стек уже используется в банке — на нём работает система кол-центра, и в команде АБС есть специалисты с опытом в Java. Сервис принимает заявки от ИБ по REST и публикует их в Kafka.

**3. OTP-модуль как расширение ИБ**

Функционал отправки СМС уже есть в ядре ИБ, но его доработка потребует участия подрядчика — а это дополнительные сроки и бюджет. Бизнес хочет этого избежать. Поэтому банковская команда реализует OTP-модуль самостоятельно как новый компонент внутри существующего .NET-решения, который напрямую обращается к СМС-шлюзу.

**4. Модуль управления ставками в АБС + ABS API Gateway**

Ставки хранятся только в Excel-файлах, недоступных для ИБ и сайта. Реализуется новый модуль управления ставками в АБС (PL-SQL + раздел в десктопном клиенте Delphi). Бизнес-логика расчёта и хранения ставок остаётся в PL-SQL, разграничение доступа между депозитным и кредитным блоками сохраняется (F14).

Однако АБС — это PL-SQL/Oracle + десктопный клиент на Delphi, у неё нет собственного REST API. Чтобы ИБ, сайт и система КЦ могли получать ставки, вводится тонкий **ABS API Gateway** на Java Spring Boot — он оборачивает PL-SQL процедуры в REST-интерфейс. Java-стек уже знаком команде АБС. ИБ кеширует полученные ставки локально, чтобы не создавать лишнюю нагрузку на АБС и обеспечить быстрый отклик.

**5. Горизонтальное масштабирование ИБ**

Сейчас ИБ работает на одной группе серверов в одном ЦОД — при сбое он просто перестаёт работать. Для обеспечения 99,9% доступности перед серверами ИБ добавляется балансировщик нагрузки (Nginx), а сам ИБ разворачивается в обоих ЦОД как активные узлы.

- Диаграмма контекста (C4 Level 1): `c4_context.puml`
- Диаграмма контейнеров (C4 Level 2): `c4_container.puml`

### **Альтернативы**

**1. Прямые синхронные вызовы ИБ → REST API АБС**

Вариант: реализовать REST API у АБС и вызывать его синхронно из ИБ.

Отклонено: БД АБС перегружена, синхронные вызовы при пиковой нагрузке нарушают R4 и создают риск недоступности всего банка. Создание REST API у АБС потребует значительных трудозатрат команды АБС и не снимает проблему нагрузки на Oracle.

**2. Прямая интеграция ИБ → Kafka без промежуточного сервиса**

Вариант: добавить Kafka-клиент непосредственно в .NET 4.5 проект ИБ.

Отклонено: текущая версия платформы ИБ несовместима с Kafka-клиентами (+R6). Потребовало бы вмешательства в ядро системы подрядчика — нежелательно и затратно (+R2).

**3. RabbitMQ вместо Kafka**

Вариант: использовать RabbitMQ, который имеет хорошую поддержку .NET Framework.

Отклонено: стратегический стандарт банка — Kafka (+R6). RabbitMQ создаст дополнительный технологический зоопарк без долгосрочных преимуществ. Промежуточный Java-сервис решает проблему совместимости без отказа от Kafka.

### **Недостатки, ограничения, риски**

| **№** | **Описание** |
| :- | :- |
| 1 | **Задержка обработки.** Асинхронная модель означает, что клиент не получит мгновенного открытия депозита — только SMS после ручной обработки бэк-офисом. Необходимо информировать клиента о статусе в UI (например, «Заявка принята, ожидайте звонка/СМС»). |
| 2 | **Новая зависимость от Kafka.** Kafka — новый компонент инфраструктуры для банка. Требует экспертизы по эксплуатации (мониторинг, репликация, резервирование). Необходимо обучение команды. |
| 3 | **Монолитный ИБ.** Горизонтальное масштабирование возможно на уровне серверов, но монолитная архитектура .NET MVC не позволяет масштабировать отдельные компоненты независимо. Это ограничение снимается только при переводе ИБ на микросервисную архитектуру в будущем. |
| 4 | **Ставки в АБС — точка отказа.** Если модуль управления ставками в АБС недоступен, ИБ и сайт не смогут показывать актуальные данные. Кеширование на стороне ИБ снижает риск, но не устраняет его полностью. |
| 5 | **Персонализированные ставки в MVP.** Логика расчёта персонализированных ставок (зависит от данных кредитного блока) требует согласования между двумя подразделениями с разграничением доступа. В рамках MVP рекомендуется ограничиться стандартными ставками, вынести персонализацию в следующую итерацию. |
