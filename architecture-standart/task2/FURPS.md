| Код | Требования (As-Is + MVP) | Комментарий |
|---|---|---|
| **F** | **Функциональные (Functionality)** | |
| F1 | **Витрина депозитов на сайте (MVP):** сайт показывает список доступных депозитов/накопительных счетов с актуальными ставками и условиями. | |
| F2 | **Заявка с сайта (MVP):** новый клиент может подать заявку на депозит, указав Ф. И. О. и номер телефона (и параметры депозита). | |
| F3 | **Маршрутизация заявки с сайта в КЦ (MVP):** заявка автоматически создаётся в системе кол-центра банка, формируется задача на исходящий звонок менеджера. | |
| F4 | **Особые условия от менеджера КЦ (MVP):** менеджер кол-центра может зафиксировать предложенные особые условия (например, спецставку/комментарий) в заявке. | |
| F5 | **Идентификация нового клиента (As-Is + MVP):** после звонка новый клиент должен прийти в отделение для очной идентификации, после чего возможно открытие депозита. | |
| F6 | **Витрина депозитов в интернет-банке (MVP):** клиент в ИБ видит список депозитов/НС с актуальными ставками. | Персонализированные ставки вынесены за рамки MVP (см. риск 5 в ADR). |
| F7 | **Заявка из интернет-банка (MVP):** клиент выбирает счёт списания и сумму, подаёт заявку на открытие депозита/НС. | |
| F8 | **Подтверждение заявки SMS-кодом (MVP):** подача заявки/открытие депозита в ИБ подтверждается одноразовым SMS-кодом (OTP). | |
| F9 | **Передача обращений/заявок КЦ → АБС (As-Is + MVP):** система КЦ передаёт зарегистрированную заявку/обращение в АБС для дальнейшей обработки. | |
| F10 | **Обработка заявки бэк-офисом в MVP:** менеджер бэк-офиса подтверждает условия депозита (в т.ч. ставку) и фиксирует их в АБС. | |
| F11 | **Открытие депозита в отделении (As-Is + MVP):** сотрудник отделения открывает депозит в АБС и оформляет документы по текущему процессу. | |
| F12 | **SMS-уведомления по статусам (As-Is + MVP):** клиент получает SMS после подтверждения ставки/условий и после открытия депозита, а также при отказе/ошибке. | |
| F13 | **Управление ставками (As-Is + MVP):** сотрудники бэк-офиса депозитов и бэк-офиса кредитов должны иметь возможность работать со ставками/коэффициентами и согласовывать их по текущему процессу, с фиксацией результата в АБС. | Сейчас ставки ведутся в Excel — нужен модуль в АБС для централизованного хранения и доступа из ИБ/сайта/КЦ. |
| F14 | **Разграничение доступа (As-Is + MVP):** депозитный и кредитный блоки не получают доступ к данным друг друга, кроме разрешённых данных по ставкам/коэффициентам. | Организационное требование банка: два блока работают в одной АБС, но с разделением прав. |
| **U** | **Удобство использования (Usability)** | |
| U1 | **Адаптивность (MVP):** витрина и форма заявки на сайте и в ИБ корректно работают на мобильных устройствах. | Значительная доля клиентов заходит с мобильных — без адаптивности теряется конверсия. |
| U2 | **Предзаполнение (MVP):** в ИБ форма заявки предзаполняется известными данными клиента. | Клиент уже авторизован в ИБ — нет смысла запрашивать ФИО и телефон повторно. |
| U3 | **Дизайн-система (MVP):** интерфейсы сайта/ИБ используют принятую в компании дизайн-систему (цвета, брендированные элементы, компоненты). | |
| **R** | **Надёжность (Reliability)** | |
| R1 | **Доступность (MVP):** сервисы интернет-банка и потоки подачи заявок должны работать 24/7 и быть доступны в 99,9% случаев. | |
| R2 | **Переключение ЦОД (As-Is + MVP):** при сбое основного ЦОД интернет-банк должен иметь возможность переключения на резервный ЦОД. | |
| R3 | **Гарантированная доставка заявок (MVP):** заявки с сайта/ИБ не должны теряться при недоступности АБС/КЦ; используется буферизация/очередь/ретраи. | Команда АБС сообщает, что БД перегружена — нужна очередь для сглаживания пиков и гарантии доставки при сбоях. |
| R4 | **Снижение нагрузки на АБС (MVP):** новая функциональность подачи заявок не должна создавать пиковую синхронную нагрузку на АБС; предпочтительно асинхронное взаимодействие и ограничение частоты запросов. | Команда АБС прямо указывает: «онлайн-подача заявок может поставить под угрозу работоспособность банка». |
| **P** | **Производительность (Performance)** | |
| P1 | **Быстрый отклик (MVP):** отклик по операциям в UI должен занимать миллисекунды; справочные данные (в т.ч. для платежей) должны загружаться без задержек более 1 секунды. | |
| **S** | **Поддерживаемость (Supportability)** | |
| S1 | **Переиспользуемость интеграций (MVP):** интеграции по заявкам должны позволять расширение на онлайн-кредитование без полной переработки. | Бизнес планирует онлайн-кредитование как следующий шаг — архитектура интеграций не должна быть одноразовой. |
| S2 | **Документация (MVP):** для новых/изменённых интерфейсов и интеграций должна быть подготовлена документация (контракты API, схемы, описание статусов, эксплуатационные заметки). | Много команд и подрядчиков — без документации сопровождение невозможно. |
| S3 | **Мониторинг и логирование (MVP):** новые сервисы (Сервис заявок, ABS API Gateway, Rate Export Service) и Kafka должны быть подключены к системе мониторинга и централизованного логирования. | Без мониторинга невозможно оперативно обнаруживать сбои в асинхронных цепочках (заявки, экспорт ставок). |
| **+R** | **Ограничения (Restrictions)** | |
| +R1 | **Шифрование трафика (MVP):** передача данных между клиентом и сайтом/ИБ, а также между системами, должна выполняться по защищённым каналам (TLS). | |
| +R2 | **Технологический контур ИБ (As-Is + MVP):** изменения в ИБ выполняются в рамках ASP.NET MVC 4.5 / .NET Framework 4.5. | |
| +R3 | **Технологический контур сайта/КЦ (As-Is + MVP):** сайт — PHP/React; система КЦ — React/Java Spring Boot/PostgreSQL. | |
| +R4 | **SMS без доработки “ядра” (MVP):** функционал подтверждения SMS-кодом и уведомлений реализуется силами команды банка без изменения ядра интернет-банка у подрядчика. | Функционал SMS реализован в ядре ИБ, но бизнес хочет избежать зависимости от подрядчика по срокам и бюджету. |
| +R5 | **Технологии и экспертиза (MVP):** приоритет использования существующих технологий банка (MS SQL, Oracle); новые технологии допускаются только при совместимости с текущими платформами и наличии экспертизы. | Снижение рисков: банк небольшой, найм специалистов по новым стекам затратен. |
| +R6 | **Очереди сообщений (MVP):** если требуется брокер сообщений, целевой стандарт — Kafka; если текущая платформа ИБ несовместима, используется совместимый промежуточный компонент/адаптер без изменения ядра ИБ. | ИБ на .NET 4.5 — нет совместимых Kafka-клиентов. Нужен адаптер. |
| +R7 | **Прямой доступ ИБ к БД АБС (As-Is):** текущая интеграция ИБ с АБС выполняется через прямые запросы к БД Oracle. | Фиксация текущего состояния: новый функционал не должен использовать этот паттерн, но старый (платежи, счета) остаётся. |
| +R8 | **Масштабирование (MVP):** предусмотреть горизонтальное масштабирование интернет-банка и балансировку нагрузки; учитывать, что АБС масштабируется преимущественно вертикально из-за БД. | Сейчас ИБ работает на одной группе серверов в одном ЦОД — это не обеспечивает 99,9% доступности. |
