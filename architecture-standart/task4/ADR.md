### **Решение**

#### Логика принятых решений

**1. Система КЦ получает ставки из АБС по REST API**

Система КЦ уже развёрнута в инфраструктуре банка на стеке Java Spring Boot / PostgreSQL. В рамках MVP депозитов в АБС появляется модуль управления ставками с REST API (через ABS API Gateway). Система КЦ вызывает этот API ежедневно по расписанию и сохраняет ставки в свою БД для кеширования. При недоступности АБС операторы работают с последними закешированными данными. Это минимальное по объёму изменение двух уже существующих систем.

**2. Партнёрский КЦ получает ставки через Rate Export Service (SFTP)**

Партнёрский КЦ работает во внешней системе и не имеет возможности принимать API-вызовы — только файлы по SFTP. Вводится новый лёгкий сервис Rate Export Service (Java Spring Boot, scheduled job), который:
- Ежедневно по расписанию запрашивает актуальные ставки из АБС Rate Management API
- Генерирует CSV-файл
- Передаёт файл на SFTP-сервер партнёрского КЦ

Сервис реализуется на Java Spring Boot — этот стек уже используется в банке, что снижает порог вхождения для команды. Решение не затрагивает внутренние компоненты АБС.

#### Диаграммы

- Диаграмма контекста (C4 Level 1): `c4_context.puml`
- Диаграмма контейнеров (C4 Level 2): `c4_container.puml`

### **Альтернативы**

**1. Экспорт ставок через Oracle Scheduler (без отдельного сервиса)**

Вариант: реализовать экспорт файла ставок прямо в АБС через Oracle Scheduler + UTL_FILE / внешний скрипт.

Отклонено: SFTP-взаимодействие из Oracle PL-SQL нетривиально, требует дополнительных настроек БД. Создание отдельного Java-сервиса чище, легче в сопровождении и не усложняет архитектуру АБС. Java-стек уже присутствует в банке (+R5).

**2. Ставки в партнёрский КЦ через Kafka + адаптер**

Вариант: партнёрский КЦ получает обновления ставок через Kafka (партнёр разворачивает Kafka-consumer).

Отклонено: партнёрский КЦ работает во внешней информационной системе и технически не может принимать API-вызовы — только файлы по SFTP. Это явное ограничение со стороны партнёра.

**3. Реального времени API ставок в системе КЦ (без кеша)**

Вариант: система КЦ делает live-запрос к АБС Rate API при каждом обращении оператора.

Отклонено: БД АБС уже перегружена, и 200+ операторов КЦ, делающих live-запросы, создадут заметную дополнительную нагрузку. Учитывая, что ставки обновляются раз в сутки, ежедневная синхронизация с кешем полностью достаточна.

### **Недостатки, ограничения, риски**

| **№** | **Описание** |
| :- | :- |
| 1 | **Задержка обновления ставок в партнёрском КЦ.** Ставки передаются раз в сутки по расписанию. В случае внепланового изменения ставки партнёрский КЦ узнает об этом не сразу. Необходимо договориться о процессе экстренного уведомления (например, e-mail дополнительно). |
| 2 | **Зависимость от SFTP-сервера партнёра.** Если SFTP партнёра недоступен, файл не доставится в срок. Ретраи снижают риск, но не устраняют полностью. Нужен мониторинг и алертинг на неудачные передачи. |
| 3 | **Ставки в КЦ на 1 день позади при сбое АБС.** Кеш в системе КЦ обновляется ежедневно. Если АБС недоступна в момент синхронизации, операторы будут работать с устаревшими ставками до следующего успешного обновления. |
