# Проверочный лист безопасности бизнес-систем PropDevelopment

## Обоснование выбора разделов

Выбор разделов проверочного листа основан на анализе описания компании,
диаграммы контейнеров C4 и выявленных проблем бизнеса.
Разделы приоритизированы по степени влияния на ключевые проблемы
PropDevelopment.

### Включённые разделы и обоснование

**I. Управление доступом** — ПРИОРИТЕТ: ВЫСОКИЙ

Обоснование: в описании компании зафиксированы критические проблемы
с контролем доступа. Партнёры из одной управляющей компании могут
видеть и изменять данные клиентов другой компании (нарушение
авторизации). Клиенты видят в личном кабинете чужие ФИО (нарушение
изоляции сессий). Несколько точек регистрации клиентов без
координации создают риск дублирования учётных записей.
Используются два экземпляра Active Directory (для финансов и для
бизнес-систем), что требует проверки согласованности политик.
Аутентификация реализована через Keycloak (auth-service-1), но нет
данных о применении MFA для критических операций (онлайн-сделки,
управление замками «Умного дома»).

**II. Безопасность данных** — ПРИОРИТЕТ: КРИТИЧЕСКИЙ

Обоснование: это ключевая проблема компании, прямо обозначенная
в описании. Нарушен контроль данных — внутренние потоки между
IT-продуктами контролируются не системно. Ограничения по данным
обусловлены только бизнес-процессами, а не техническими мерами.
Системы обработки данных (DWH на Greenplum) не имеют ограничений
на доступ к данным. CDC-потоки из 5 баз данных передают сырые
данные в хранилище без маскирования. Компания обрабатывает
персональные данные (ФИО, телефоны, email, паспортные данные для
онлайн-сделок, платёжные данные ЖКУ), но нет уверенности в
соответствии ФЗ-152. Проблемы с данными блокируют развитие
BI/ML/AI-продуктов.

**III. Безопасность API и интеграций** — ПРИОРИТЕТ: КРИТИЧЕСКИЙ

Обоснование: отдельный раздел по API необходим, так как это одна
из двух ключевых проблем, явно описанных в задании. Системы
PropDevelopment используют API управляющих компаний без единых
политик безопасности. Контракты API содержат категории данных,
которые явно предоставляют персональные данные. Обнаружены
уязвимости типа IDOR — партнёр одной УК может получить доступ
к данным другой УК. Компания интегрирована с госорганами через
СМЭВ (SOAP), платёжной системой и множеством поставщиков ЖКХ.
При планах расширения партнёрских сервисов эти проблемы будут
масштабироваться.

**IV. Защита инфраструктуры** — ПРИОРИТЕТ: СРЕДНИЙ

Обоснование: компания использует гибридную инфраструктуру —
собственный ЦОД, приватное облако и арендованные мощности у
облачного провайдера. Команды могут временно использовать
облачные ресурсы для доработки и тестирования, что расширяет
периметр атаки. На диаграмме указаны Firewall на входе для
внешних пользователей, но нет данных о сегментации внутренней
сети между доменами (продажи, ЖКУ, финансы, дата). Приоритет
средний, так как базовые меры (Firewall, приватное облако)
уже реализованы.

**V. Управление инцидентами** — ПРИОРИТЕТ: ВЫСОКИЙ

Обоснование: недавно у конкурента произошла утечка данных,
активно обсуждавшаяся в СМИ. Руководство обеспокоено и хочет
предотвратить подобные инциденты. При этом текущий аудит — первый
комплексный аудит безопасности компании, что указывает на
отсутствие системной работы с инцидентами. Единственный специалист
по ИБ получает не все решения и изменения от команд из-за
нарушенной коммуникации после смены руководства.

**VI. Организация процессов ИБ** — ПРИОРИТЕТ: ВЫСОКИЙ

Обоснование: организационные проблемы являются корневой причиной
многих технических уязвимостей. Специалист по ИБ один на всю
компанию (4 домена, множество продуктовых команд). После смены
руководства IT-команд нарушена передача информации — ИБ-специалист
не получает все решения и изменения. В структуре продуктовых
команд нет выделенной роли по безопасности. DevOps-инженер есть,
но security-чемпион — нет. Без решения организационных проблем
технические меры будут неэффективны.

**VII. Соответствие законодательству (Compliance)** — ПРИОРИТЕТ: ВЫСОКИЙ

Обоснование: одна из целей аудита — проверить соответствие
требованиям законодательства РФ и предотвратить штрафы.
IT-ландшафт активно развивался, появлялись новые интеграции,
и нет уверенности, что все бизнес-процессы соответствуют нормам.
Компания обрабатывает ПДн клиентов и собственников, проводит
финансовые операции (оплата ЖКУ, онлайн-сделки), интегрируется
с госорганами. Всё это подпадает под ФЗ-152, ФЗ-149 и
требования регуляторов.

### Не включённые разделы и обоснование

**Физическая безопасность** — не включён, так как в описании нет
данных о физическом доступе к серверам и ЦОД. Это важный раздел,
но для его проверки недостаточно информации в текущем контексте.
Рекомендуется включить в расширенный аудит.

**Безопасность разработки (SDLC)** — не включён как отдельный
раздел, так как в описании недостаточно данных о процессах
разработки. Частично покрывается разделом «Организация процессов
ИБ». Рекомендуется включить при расширении аудита.

**Обучение и осведомлённость сотрудников** — не включён как
отдельный раздел из-за недостатка данных. Частично покрывается
разделом «Управление инцидентами».

| 2 | Обеспечена ли коммуникация между ИБ-специалистом и всеми продуктовыми командами? | **Нет** | Прямо указано: после изменений в составе руководителей IT-команд нарушился процесс передачи информации. ИБ-специалисту передают не все решения и изменения. |
| 3 | Включён ли ИБ-специалист в процессы принятия архитектурных решений? | **Нет** | Следует из нарушения коммуникации: ИБ-специалист не получает информацию о решениях и изменениях. |
| 4 | Есть ли в продуктовых командах роли, ответственные за безопасность (security champions)? | **Нет** | Состав команды: владелец продукта, бизнес-аналитики, разработчики, инженеры по эксплуатации, DevOps-инженер. Роли по безопасности нет. |
| 5 | Разработана и утверждена ли общая политика информационной безопасности компании? | **Нет** | Системы работают без единых политик безопасности (подтверждено для API партнёров). Отсутствие общей политики — корневая причина многих проблем. |
| 6 | Проводятся ли регулярные review безопасности при внедрении новых интеграций и сервисов? | **Нет** | Команды разработки самостоятельно обнаруживают уязвимости в API партнёров, что означает отсутствие формального процесса security review. |
| 7 | Доводятся ли требования ИБ до всех продуктовых и операционных команд? | **Нет** | Нарушена передача информации между командами. ИБ-специалист не может эффективно транслировать требования при текущей коммуникации. |
| **VII. Соответствие законодательству (Compliance)** | | | |
| 1 | Определён ли перечень применимых нормативных актов (ФЗ-152, ФЗ-149 и др.)? | Нет данных | Компания не уверена, что все бизнес-процессы соответствуют законодательным нормам |
| 2 | Зарегистрирована ли компания как оператор персональных данных в реестре Роскомнадзора? | Нет данных | Компания обрабатывает ПДн клиентов и собственников в нескольких системах — регистрация обязательна по ФЗ-152. |
| 3 | Получено ли согласие субъектов на обработку персональных данных? | Нет данных | Клиенты предоставляют ФИО, телефон, email при бронировании и расширенную форму при онлайн-сделке. Собственники используют мобильное приложение. |
| 4 | Соблюдается ли принцип минимизации обработки ПДн (обрабатываются только необходимые данные)? | **Нет** | API партнёров передают избыточные категории ПДн. CDC-потоки копируют все данные из БД в DWH без фильтрации. |
| 5 | Обеспечена ли локализация хранения ПДн российских граждан на территории РФ? | Нет данных | Компания использует гибридную инфраструктуру с облачным провайдером. Необходимо проверить расположение серверов. |
| 6 | Реализованы ли организационные и технические меры по защите ПДн согласно требованиям ФСТЭК? | Нет данных | Учитывая выявленные проблемы (отображение чужих данных, IDOR в API)|
| 7 | Назначено ли ответственное лицо за организацию обработки ПДн? | Нет данных | Формально может быть ИБ-специалист, но он один на всю компанию и изолирован от процессов. |
| 8 | Соответствует ли интеграция с госорганами (СМЭВ) требованиям регуляторов? | Нет данных | client-mart-app интегрирован с госорганами через СМЭВ (SOAP). Необходимо проверить соответствие требованиям Минцифры. |
