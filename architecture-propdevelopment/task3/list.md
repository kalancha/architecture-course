# Требования к безопасности внешних интеграций «Умный дом»

## I. Защита данных

| ID | Требование | Приоритет | Обоснование |
|:---|:---|:---|:---|
| SEC-1 | Все данные при передаче между PropDevelopment и платформой партнёра должны быть зашифрованы с использованием TLS 1.3. Использование более ранних версий TLS запрещено | Критический | Передаются биометрические данные и команды управления физическим доступом в здания. Компрометация канала создаёт угрозу физической безопасности жильцов |
| SEC-2 | Биометрические данные (шаблоны лиц) должны передаваться в виде математических хэшей/векторов, а не исходных фотографий. Исходные фотографии не должны покидать устройство собственника | Критический | ФЗ-152, ст. 11: биометрические ПДн — специальная категория, требующая повышенной защиты. Передача raw-фото создаёт неприемлемый риск утечки необратимых данных |
| SEC-3 | Номера автомобилей должны передаваться в хэшированном виде для сопоставления. Хранение и передача в открытом виде запрещены | Высокий | Минимизация передаваемых ПДн. Платформа партнёра выполняет сравнение хэшей, а не открытых номеров |
| SEC-4 | Запрещена передача ПДн собственников (ФИО, телефон, email, адрес) на платформу партнёра. Для идентификации используются только внутренние идентификаторы PropDevelopment | Критический | Устранение выявленной проблемы — контракты API партнёров содержат категории данных, которые явно предоставляют ПДн. Партнёру достаточно внутреннего идентификатора |
| SEC-5 | Биометрические шаблоны должны храниться в изолированном хранилище (biometry-vault) с шифрованием at-rest (AES-256). Доступ к хранилищу — только через выделенный сервис biometry-adapter | Критический | ФЗ-152: хранение биометрии требует усиленной защиты. Изоляция хранилища ограничивает поверхность атаки и blast radius при компрометации |
| SEC-6 | Журнал событий (проходы, проезды, звонки, открытия) должен храниться на стороне PropDevelopment не менее 90 дней с защитой от модификации | Высокий | Необходимо для расследования инцидентов физической безопасности. Партнёр не должен быть единственным источником правды |
| SEC-7 | Партнёр не должен хранить биометрические данные после завершения сопоставления. Максимальное время хранения в оперативной памяти — 60 секунд. Факт удаления должен подтверждаться в ответе API | Критический | Минимизация хранения биометрических ПДн у третьих лиц в соответствии с принципами ФЗ-152 |
| SEC-8 | Данные CDC-потока из smart-home-db в DWH не должны содержать биометрических данных. Передаются только обезличенные события (device_id, event_type, timestamp) | Высокий | Разделение аналитического контура и контура чувствительных данных. Предотвращение утечки биометрии через DWH |
| SEC-9 | При передаче списков доступа (whitelist номеров, список разрешённых лиц) партнёру должны передаваться только хэши/векторы и internal_user_id. Привязка к реальным ПДн остаётся только на стороне PropDevelopment | Критический | Даже при компрометации платформы партнёра невозможно сопоставить данные с реальными людьми |
| SEC-10 | Все криптографические ключи, используемые для интеграции (API-ключи, ключи подписи webhook, сертификаты mTLS), должны храниться в защищённом хранилище секретов (Vault) и не должны присутствовать в коде или конфигурационных файлах | Критический | Предотвращение утечки секретов. В текущей архитектуре отсутствует упоминание vault/HSM — это необходимо исправить |

## II. Контроль доступа

| ID | Требование | Приоритет | Обоснование |
|:---|:---|:---|:---|
| ACC-1 | Каждый ЖК должен быть изолирован на уровне API: запросы от устройств и пользователей одного ЖК не должны возвращать данные другого ЖК (tenant isolation). Параметр tenant_id обязателен в каждом запросе и должен валидироваться на обеих сторонах | Критический | Устранение выявленной проблемы IDOR — партнёр одной управляющей компании может видеть и изменять данные клиентов и ЖК другой компании |
| ACC-2 | Собственник может управлять только устройствами, привязанными к его квартире, подъезду и ЖК. Привязка верифицируется через CRM собственников при каждом запросе на управление | Критический | Принцип минимальных привилегий. Предотвращение несанкционированного открытия дверей и шлагбаумов чужих ЖК |
| ACC-3 | Команды открытия двери и шлагбаума должны требовать подтверждение авторизации для каждого запроса (no caching of authorization decisions). Кэширование решений авторизации для write-операций запрещено | Критический | Управление физическим доступом в здания требует актуальной проверки прав. Кэширование может привести к открытию двери пользователем, у которого права уже отозваны |
| ACC-4 | API-ключи партнёра должны быть уникальными для каждого ЖК (tenant_id) и ротироваться не реже 1 раза в 90 дней. Должен быть реализован механизм автоматической ротации | Высокий | Решение выявленной проблемы отсутствия единых политик по API-ключам партнёров. Уникальность ключей по ЖК ограничивает blast radius при компрометации |
| ACC-5 | Администратор управляющей компании может управлять только устройствами и списками доступа своей УК. Разделение реализуется через scope в OAuth-токене (uk_id) | Высокий | Ограничение области ответственности. Администратор УК не должен влиять на устройства, не относящиеся к его компании |
| ACC-6 | Для операций управления доступом (открытие двери, управление биометрией, изменение списков доступа) обязательна многофакторная аутентификация (MFA) через TOTP или push-уведомление | Критический | Управление физическим доступом и биометрией — критические операции. Компрометация пароля не должна давать возможность открывать двери |
| ACC-7 | Регистрация и удаление биометрических шаблонов требует явного согласия собственника на обработку биометрических ПДн. Согласие должно фиксироваться в системе с timestamp и быть отзывным | Критический | Требование ФЗ-152, ст. 11: обработка биометрических ПДн допускается только с письменного согласия субъекта. Согласие должно быть отзывным |
| ACC-8 | При отзыве согласия на обработку биометрии или при удалении учётной записи собственника все биометрические шаблоны должны быть удалены из biometry-vault и с платформы партнёра в течение 24 часов | Критический | Требование ФЗ-152: прекращение обработки ПДн при отзыве согласия. Должен быть подтверждённый процесс удаления |
| ACC-9 | Доступ к biometry-vault разрешён только сервису biometry-adapter. Прямой доступ к хранилищу из других сервисов, включая smart-home-core-app и DWH, запрещён на сетевом уровне | Критический | Принцип минимальных привилегий для специальной категории ПДн. Единственная точка доступа упрощает аудит и контроль |

## III. Аутентификация и авторизация между системами

| ID | Требование | Приоритет | Обоснование |
|:---|:---|:---|:---|
| AUTH-1 | Аутентификация собственников осуществляется через существующий auth-service-1 (Keycloak) с использованием OAuth 2.0 + OpenID Connect. Для мобильного приложения используется flow Authorization Code + PKCE | Критический | Использование существующего IdP, стандартный безопасный flow для мобильных приложений. PKCE защищает от перехвата authorization code |
| AUTH-2 | Для доступа к функциям «Умного дома» в Keycloak должен быть создан отдельный realm или набор scopes: smarthome:intercom:view, smarthome:intercom:open, smarthome:barrier:open, smarthome:access:manage, smarthome:biometry:manage, smarthome:events:view | Высокий | Гранулярное разделение прав. Собственник может иметь доступ к просмотру видео домофона, но не к управлению биометрией |
| AUTH-3 | Взаимодействие smart-home-core-app с платформой партнёра осуществляется через API Gateway с использованием OAuth 2.0 Client Credentials для machine-to-machine аутентификации и mTLS (mutual TLS) для двусторонней аутентификации | Критический | Стандартизация M2M-аутентификации. mTLS обеспечивает подтверждение идентичности обеих сторон на уровне транспорта |
| AUTH-4 | Все запросы от smart-home-core-app к платформе партнёра должны подписываться HMAC-SHA256 с использованием секретного ключа, уникального для каждого tenant_id | Высокий | Защита от подмены запросов (replay attack, MITM). Уникальность ключа по tenant ограничивает ущерб при компрометации |
| AUTH-5 | Callbacks (webhooks) от платформы партнёра должны: (a) быть подписаны HMAC-SHA256 с ключом, уникальным для tenant_id; (b) приниматься только с IP-адресов из whitelist партнёра; (c) содержать tenant_id, который валидируется на стороне PropDevelopment | Критический | Защита от поддельных callback'ов. Без проверки подписи злоумышленник может отправить ложное событие «дверь открыта» или инициировать открытие |
| AUTH-6 | Авторизация на smart-home-core-app реализуется по модели ABAC (Attribute-Based Access Control) с проверкой атрибутов: user.tenant_id == device.tenant_id, user.building_id == device.building_id, user.role ∈ {owner, admin_uk} | Критический | ABAC обеспечивает гибкую и точную проверку доступа на основе контекста. Предотвращает доступ собственника одного ЖК к устройствам другого |
| AUTH-7 | Взаимодействие между внутренними сервисами (smart-home-core-app → biometry-adapter) осуществляется с использованием service-to-service JWT-токенов с ограниченным временем жизни (TTL ≤ 5 минут) | Высокий | Аутентификация между микросервисами. Короткий TTL минимизирует риск при компрометации токена |
| AUTH-8 | Аутентификация администраторов УК осуществляется через Active Directory с федерацией через Keycloak. MFA обязательна. Scope ограничен идентификатором управляющей компании (uk_id) | Высокий | Использование существующей инфраструктуры AD. Федерация через Keycloak обеспечивает единую точку аутентификации |
| AUTH-9 | JWT-токены для операций «Умного дома» должны содержать claims: tenant_id (ЖК), building_id (дом), unit_id (квартира), role, scopes. Время жизни access token — не более 15 минут | Высокий | Контекстная информация в токене позволяет smart-home-core-app принимать решения по авторизации без дополнительных запросов. Короткий TTL снижает риск |

## IV. Мониторинг и аудит

| ID | Требование | Приоритет | Обоснование |
|:---|:---|:---|:---|
| MON-1 | Все команды открытия двери и шлагбаума должны логироваться с указанием: timestamp, device_id, user_id (internal), action, result (success/failure), source (app/biometry/plate/manual), IP-адрес источника запроса | Критический | Журнал физического доступа критичен для расследования инцидентов безопасности. Полнота записи обеспечивает восстановление картины события |
| MON-2 | Все вызовы API партнёра должны логироваться на API Gateway: request/response (без чувствительных данных), latency, HTTP-статус, tenant_id, correlation_id | Высокий | Решение выявленной проблемы отсутствия логирования API-взаимодействий с партнёрами. Необходимо для аудита и troubleshooting |
| MON-3 | Система должна генерировать автоматические алерты при: (a) более 5 неуспешных попыток открытия двери за 10 минут с одного устройства; (b) более 10 нераспознанных лиц за 30 минут; (c) попытке доступа к данным чужого tenant; (d) аномальном количестве API-вызовов | Высокий | Раннее обнаружение атак: перебор замков, попытки подбора биометрии, попытки эксплуатации IDOR, DDoS на устройства |
| MON-4 | Rate limiting на API Gateway: не более 10 команд открытия в минуту на устройство, не более 100 запросов в минуту на ЖК, не более 1000 запросов в минуту суммарно на партнёра | Высокий | Защита от brute-force атак и злоупотреблений API. Лимиты на уровне устройства предотвращают массовые попытки открытия |
| MON-5 | Все операции с биометрическими данными (регистрация, удаление, сопоставление) должны логироваться в отдельный защищённый журнал с невозможностью модификации (append-only) | Критический | Требование для аудита обработки специальной категории ПДн. Необходимо для подтверждения соответствия ФЗ-152 |
| MON-6 | Ежемесячный автоматизированный отчёт по безопасности интеграции: количество инцидентов, аномалий, статистика по типам доступа, SLA партнёра. Отчёт направляется ИБ-специалисту | Средний | Решение проблемы изоляции ИБ-специалиста. Автоматический отчёт обеспечивает информирование даже при нарушенной коммуникации |

## V. Требования к API-контракту партнёра

| ID | Требование | Приоритет | Обоснование |
|:---|:---|:---|:---|
| API-1 | Все endpoint'ы должны поддерживать параметр tenant_id (ЖК) и валидировать его на стороне партнёра. Запросы без tenant_id или с невалидным tenant_id должны отклоняться с HTTP 403 | Критический | Tenant isolation — устранение выявленной проблемы IDOR |
| API-2 | Ответы API не должны содержать ПДн собственников. Допустимые поля в ответах: internal_user_id, device_id, event_type, timestamp, result | Критический | Минимизация данных в контрактах API. Устранение выявленной проблемы — контракты содержат ПДн |
| API-3 | API должен поддерживать OAuth 2.0 Client Credentials для machine-to-machine аутентификации | Критический | Стандартизация аутентификации для M2M-взаимодействия |
| API-4 | Webhooks должны подписываться HMAC-SHA256 с ключом, уникальным для каждого tenant_id. Формат подписи: HMAC-SHA256(secret, timestamp + body) | Критический | Защита от подмены callback'ов от партнёра |
| API-5 | API должен поддерживать endpoint для отзыва биометрических шаблонов (DELETE /biometry/{template_id}) с подтверждением удаления в ответе | Критический | Возможность оперативного реагирования на инциденты и выполнение требований ФЗ-152 по отзыву согласия |
| API-6 | Время ответа API партнёра для команд управления устройствами (open door, open barrier) не должно превышать 3 секунды (p99) | Высокий | Домофон и шлагбаум — realtime-сценарии. Длительное ожидание неприемлемо для пользователя и может создать проблемы безопасности (собственник стоит у закрытой двери) |
| API-7 | Партнёр обязан обеспечить SLA не ниже 99.9% для endpoint'ов управления устройствами. Плановые работы — не более 4 часов в месяц с уведомлением за 48 часов | Высокий | Доступность сервиса управления физическим доступом = безопасность жильцов. Невозможность открыть дверь — критический инцидент |
| API-8 | Версионирование API (v1, v2 и т.д.). Минимум 6 месяцев поддержки предыдущей версии после выпуска новой. Уведомление о deprecation за 3 месяца | Средний | Предсказуемость изменений, возможность планирования миграции без экстренных доработок |
| API-9 | API должен поддерживать endpoint для получения статуса устройства (GET /devices/{device_id}/status) с информацией: online/offline, last_seen, firmware_version | Средний | Мониторинг состояния устройств для проактивного обнаружения проблем |
| API-10 | Партнёр обязан уведомлять PropDevelopment об инцидентах безопасности на своей стороне в течение 24 часов с момента обнаружения | Критический | Оперативное реагирование на инциденты в цепочке поставки. PropDevelopment должен иметь возможность отключить интеграцию или отозвать данные |

## VI. Отказоустойчивость

| ID | Требование | Приоритет | Обоснование |
|:---|:---|:---|:---|
| FT-1 | При недоступности платформы партнёра: команды открытия через приложение — retry (3 попытки, exponential backoff), затем уведомление собственнику «Сервис временно недоступен». Автоматическое открытие по биометрии — локальный кэш шаблонов на устройстве (если поддерживается) | Высокий | Обеспечение базовой функциональности при сбоях партнёра. Собственник не должен остаться заблокированным |
| FT-2 | При недоступности API Gateway — circuit breaker на smart-home-core-app. Команды ставятся в очередь (Kafka) с TTL=30 секунд для команд управления | Высокий | Graceful degradation. Команды управления имеют короткий TTL, так как устаревшая команда на открытие двери может быть небезопасна |
| FT-3 | При недоступности Keycloak — кэширование валидации JWT на smart-home-core-app (TTL=5 минут) только для read-операций (просмотр видео, журнал событий). Write-операции (открытие двери, управление биометрией) требуют актуальной валидации | Критический | Баланс между доступностью и безопасностью. Операции управления физическим доступом не могут полагаться на кэшированную аутентификацию |
| FT-4 | При недоступности biometry-vault — регистрация новой биометрии невозможна. Существующие шаблоны, уже переданные партнёру, продолжают работать. Собственник информируется о временной недоступности функции | Средний | Деградация затрагивает только регистрацию новой биометрии, не влияет на текущий доступ |
| FT-5 | Должен быть реализован механизм экстренного отключения интеграции с партнёром (kill switch) без влияния на остальные функции мобильного приложения собственников | Критический | Возможность оперативной реакции при обнаружении компрометации платформы партнёра или массовой уязвимости |
