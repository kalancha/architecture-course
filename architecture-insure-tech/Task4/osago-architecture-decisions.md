# Task4. Проектирование продажи ОСАГО

## Основная идея

Оформление ОСАГО - долгий асинхронный процесс. Для одного пользователя нужно отправить заявку во все страховые компании, дождаться предложений и показывать их сразу после получения. При 2,5 тыс. одновременных пользователей и 10 страховых компаниях система может получить до 25 тыс. параллельных внешних сценариев.

Поэтому не держим синхронный REST-запрос `core-app -> osago-aggregator` до 60 секунд. Вместо этого комбинируем подходы: frontend получает идентификатор заявки синхронно, а дальнейшая обработка идёт асинхронно через брокер сообщений.

## osago-aggregator

`osago-aggregator` отвечает только за техническую интеграцию со страховыми компаниями:

- отправляет заявки в страховые компании;
- хранит соответствие `applicationId -> insurerRequestId`;
- опрашивает страховые компании до получения предложения или истечения 60 секунд;
- отправляет в `core-app` событие о каждом полученном предложении.

## Нужно ли своё хранилище

Да, `osago-aggregator` нужно своё хранилище.

Причины:

- сервис развёрнут в нескольких экземплярах, им нужен общий источник состояния;
- нужно хранить идентификатор заявки, который вернула каждая страховая компания;
- нужны статусы обработки, время ожидания до 60 секунд и количество повторных попыток;
- нужно исключить двойную обработку одной и той же заявки разными репликами.

## API osago-aggregator для core-app

Основной API - асинхронный через брокер:

- команда `CreateOsagoApplication` от `core-app` к `osago-aggregator`;
- событие `OsagoOfferReceived` от `osago-aggregator` к `core-app`;
- событие `OsagoOfferFailed`, если страховая компания не ответила за 60 секунд;
- событие `OsagoApplicationCompleted`, когда ответы получены или истёк общий лимит ожидания.

Синхронный REST API можно оставить только для служебных операций

## Интеграция core-app и osago-aggregator

Используем Kafka:

1. `core-app` сохраняет заявку ОСАГО и через Transactional Outbox отправляет команду `CreateOsagoApplication`.
2. `osago-aggregator` читает команду, создаёт задачи по страховым компаниям и начинает polling.
3. На каждый ответ страховой компании `osago-aggregator` публикует `OsagoOfferReceived`.
4. `core-app` потребляет события, сохраняет предложения и отправляет их пользователю через Server Push.

## API web application в core-app

- `POST /api/osago/applications` - создать заявку ОСАГО.
- `GET /api/osago/applications/{applicationId}` - получить текущее состояние заявки.
- `GET /api/osago/applications/{applicationId}/offers` - получить уже полученные предложения.
- `GET /api/osago/applications/{applicationId}/offers/stream` - SSE-поток для получения предложений сразу после прихода.

SSE выбран вместо WebSocket, потому что поток в основном однонаправленный: сервер отправляет предложения браузеру. При разрыве соединения браузер может переподключиться и добрать состояние через `GET /offers`.

## Несколько экземпляров сервисов

Решение не должно зависеть от того, к какой реплике `core-app` подключён пользователь.

Для этого:

- заявки и предложения сохраняются в БД `core-app`;
- результат обработки хранится во внешнем хранилище, чтобы ответ мог обработать любой экземпляр `core-app`;
- SSE поддерживает автоматическое переподключение;
- `osago-aggregator` хранит статусы задач в своей БД, чтобы после перезапуска продолжить обработку;
- повторная отправка запроса в страховую не должна создавать дубли; для этого используем ключ `applicationId + insurerId`.

## Паттерны отказоустойчивости

- Rate Limiting: на входе `POST /api/osago/applications`, а также отдельно на интеграцию с каждой страховой компанией.
- Timeout: общий лимит ожидания 60 секунд на заявку, короткие timeout-ы на каждый HTTP-вызов к страховой.
- Retry: только для временных ошибок и только пока не истёк общий лимит ожидания 60 секунд.
- Circuit Breaker: отдельно по каждой страховой компании, чтобы деградация одного внешнего API не ломала весь агрегатор.
- Bulkhead: отдельные очереди или пулы обработки по страховым компаниям, чтобы медленная компания не забрала все ресурсы.
