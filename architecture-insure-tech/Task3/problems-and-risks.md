# Task3. Проблемы и риски текущей архитектуры

## Текущая схема взаимодействий

Сейчас `core-app` и `ins-comp-settlement` получают продукты и тарифы через REST API сервиса `ins-product-aggregator`. При запросе `ins-product-aggregator` синхронно обращается ко всем страховым компаниям, агрегирует ответы и возвращает единый список.

Дополнительно `ins-comp-settlement` раз в сутки вызывает `core-app` по REST API, чтобы получить список оформленных за день страховок для взаиморасчётов.

## Проблемы

1. Синхронная зависимость от страховых компаний.

   Если хотя бы одна страховая компания отвечает медленно или с ошибкой, это увеличивает время ответа `ins-product-aggregator` и может ломать сценарии `core-app` и `ins-comp-settlement`.

2. Рост числа интеграций ухудшит задержки.

   Сейчас страховых компаний пять, скоро станет десять. При синхронной агрегации общее время ответа будет зависеть от самого медленного внешнего API, а вероятность частичных отказов вырастет.

3. Polling создаёт лишнюю нагрузку.

   `core-app` опрашивает `ins-product-aggregator` каждые 15 минут, `ins-comp-settlement` - раз в сутки. Если потребителей станет больше, они начнут дублировать одинаковые запросы.

4. Локальные реплики данных могут расходиться.

   `core-app` и `ins-comp-settlement` хранят локальные копии продуктов и тарифов, но обновляют их с разной периодичностью. Это создаёт риск работы с разными версиями тарифов.

5. Нет буферизации и асинхронной обработки.

   REST-вызовы не дают удобного механизма накопить сообщения и обработать их позже. При сбоях приходится реализовывать повторные попытки вручную в каждом сервисе.

6. Суточная выгрузка оформленных страховок через REST уязвима.

   Если ночной вызов `ins-comp-settlement -> core-app` завершится ошибкой, взаиморасчёты могут быть неполными. Повторный вызов большого списка создаёт риск дублей и дополнительную нагрузку на `core-app`.

7. Сложнее расследовать инциденты.

   В текущей схеме нет единого журнала доменных фактов: когда изменился тариф, когда была оформлена страховка, когда событие было доставлено потребителю.

## Предлагаемое решение

Добавить брокер событий, например Kafka.

Переделать на события следующие взаимодействия:

- `ins-product-aggregator -> core-app`: публикация `ProductUpdated`, `TariffUpdated`, `ProductDisabled`.
- `ins-product-aggregator -> ins-comp-settlement`: те же события о продуктах и тарифах, чтобы settlement обновлял свою локальную реплику данных без REST polling.
- `core-app -> ins-comp-settlement`: публикация `PolicyCreated`, `PolicyIssued`, `PolicyCancelled`, `PolicyRefunded` вместо ночной REST-выгрузки.

## Transactional Outbox

Паттерн Transactional Outbox нужен там, где сервис одновременно меняет свою базу и должен гарантированно опубликовать событие.

Применяем:

- в `core-app` для событий оформленных страховок, потому что запись заявки/полиса и событие `PolicyIssued` должны быть атомарны;
- в `ins-product-aggregator`, если он сохраняет у себя нормализованную копию продуктов и тарифов перед публикацией событий.

Polling publisher читает таблицу Outbox и публикует события в Kafka. Потребители должны быть идемпотентными: повторная доставка события не должна ломать локальную реплику данных.

## Дополнительные меры

- Описывать контракты событий и версионировать их, потому что Outbox-таблица фактически становится API сервиса.
- Мониторить доступность брокера, ошибки публикации событий и отставание потребителей.
- Разделять события по топикам: отдельно изменения продуктов и тарифов, отдельно события оформленных страховок.
- Использовать Bulkhead на уровне брокера: разные топики/очереди для разных потоков событий, чтобы проблемы одного потока не блокировали остальные.
