# Task1. Технологическая архитектура To-Be

Диаграмма `InureTech_технологическая архитектура_to-be.drawio` использует встроенные SVG-объекты из скачанных библиотек Yandex Cloud для draw.io. Сами библиотеки оставлены рядом в `drawio-libraries/`, чтобы было видно, откуда взята нотация.

## Ключевое решение

Для InsureTech выбрана отказоустойчивая мультизональная архитектура:

- статические файлы веб-приложения отдаются через CDN, чтобы пользователи из разных регионов получали страницы с близким временем загрузки;
- внешний трафик проходит через GSLB/DNS, API Gateway или NGINX и балансировщик нагрузки с health checks;
- backend-сервисы развёрнуты в нескольких зонах доступности в режиме Active-Active;
- для B2B API на уровне NGINX/API Gateway применяется Rate Limiting по идентификатору партнёра;
- PostgreSQL работает в схеме Primary + Standby: основной узел принимает запись, резервный узел готов принять роль primary при сбое;
- за переключение PostgreSQL на standby отвечает DB failover manager, который отслеживает состояние primary и запускает promote standby при аварии;
- настроены репликация и регулярные резервные копии, чтобы выполнить RPO 15 минут.

## Почему независимые Kubernetes-кластеры

Я выбираю независимые кластеры в разных зонах, а не один растянутый кластер, тк отказ одной зоны или сетевой сбой не затрагивает всё окружение сразу. Цена решения выше: нужно синхронизировать деплой и настройки входящего трафика между кластерами, но для требований RTO 45 минут, RPO 15 минут и доступности 99,9% такая изоляция надёжнее.

## Масштабирование

Stateless-сервисы масштабируются горизонтально через Kubernetes HPA. Это подходит для `core-app`, `client-info`, `ins-product-aggregator` и `ins-comp-settlement`, потому что HPA регулирует количество pod-ов в зависимости от метрик нагрузки.

Для PostgreSQL основная стратегия - репликация и фейловер на резервный узел. На схеме это отражено через DB failover manager: он не является отдельным бизнес-сервисом, а показывает механизм автоматического переключения primary-роли на standby. Шардирование не применяется: текущий объём данных 50 GB, а заявленная нагрузка не требует распределения данных по шардам. Шардирование усложнит транзакции, отчётность и эксплуатацию, поэтому его стоит отложить до появления реальных признаков нехватки одного кластера БД.

## RTO и RPO

RTO 45 минут закрывается автоматическим failover на уровне GSLB/балансировщиков, Kubernetes и переключения БД на Standby. RPO 15 минут закрывается репликацией БД и резервным копированием с интервалом, который не превышает 15 минут.
