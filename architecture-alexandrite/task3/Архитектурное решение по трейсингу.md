# Архитектурное решение по трейсингу

## 1. Анализ системы: где может возникнуть проблема с заказом

### Путь заказа через систему

```
[Клиент/API] → Shop API → Shop DB
                  ↓ (upload)
              3D File Storage (S3)
                  ↓ (RabbitMQ или HTTP)
              MES API → MES DB
                  ↓ (RabbitMQ)
              CRM API → Shop DB
                  ↓ (RabbitMQ)
              MES API → MES DB
                  ↓ (RabbitMQ)
              CRM API
```

### Критические точки, требующие покрытия трейсингом

| # | Точка | Что может сломаться | Система |
|---|-------|---------------------|---------|
| 1 | **Shop API → RabbitMQ** | Сообщение не отправлено или отправлено, но не персистентно. Заказ записан в Shop DB, но MES никогда не узнает о нём | Shop API, RabbitMQ |
| 2 | **RabbitMQ → MES API** (получение заказа на расчёт стоимости) | Consumer не обработал сообщение. Сообщение потеряно или застряло в очереди | RabbitMQ, MES API |
| 3 | **MES API: расчёт стоимости** | Длительная операция (2–30 мин). Инстанс падает — расчёт теряется. Очередь на расчёт растёт — заказы зависают | MES API |
| 4 | **MES API → RabbitMQ** | Расчёт завершён, результат записан в MES DB, но сообщение в CRM не отправлено | MES API, RabbitMQ |
| 5 | **RabbitMQ → CRM API** (получение для подтверждения производства) | Consumer CRM не обработал, заказ зависает | RabbitMQ, CRM API |
| 6 | **CRM API → RabbitMQ → MES API** (подтверждение → начало производства) | Подтверждение в CRM есть, но MES не получил команду на старт. Оператор не видит заказ | CRM API, RabbitMQ, MES API |
| 7 | **Внешний API → MES API** (B2B-заказы) | Запрос от партнёра принят, но не обработан, или обработан но потерян при отправке в CRM. Партнёр не получает обратную связь | MES API |
| 8 | **Загрузка 3D-файла: Shop API / MES API → S3** | Файл не загрузился, расчёт стоимости невозможен, заказ зависает | Shop API / MES API, S3 |

**Системы, которые необходимо покрыть трейсингом:**
- **Shop API** — точка входа B2C-заказов
- **MES API** — точка входа B2B-заказов, расчёт стоимости, производственные статусы
- **CRM API** — подтверждение производства, закрытие заказов
- **RabbitMQ** — передача контекста трассировки
- **S3 (3D File Storage)** — как внешний вызов из Shop API / MES API

### Данные, попадающие в трейсинг

Каждый span должен содержать:
- **trace_id** — сквозной идентификатор, связывающий все операции в рамках одного заказа
- **span_id**, **parent_span_id** — для построения дерева вызовов
- **order_id** — идентификатор заказа (бизнес-ключ для поиска)
- **order_source** — `web` или `api` (канал поступления заказа)
- **service_name** — какой сервис создал span
- **operation_name** — что делает span (`http.request`, `rabbitmq.publish`, `rabbitmq.consume`, `price.calculate`, `s3.upload`)
- **status** — успех/ошибка
- **duration** — длительность операции
- **order_status_transition** — переход статуса, если span связан со сменой статуса (`SUBMITTED → PRICE_CALCULATED`)
- **error details** — при ошибке: тип исключения, сообщение
- **client_id** — для B2B-запросов: идентификатор партнёра

## 2. Мотивация

### Зачем нужен трейсинг

Мониторинг показывает агрегированную картину: RPS, error rate, latency. Помогает следить за состоянием системы в целом. Но когда менеджер звонит и говорит «заказ №12345 клиента потерялся», мониторинг не может сказать, где именно этот конкретный заказ застрял.

Трейсинг отвечает на вопрос «что произошло с конкретным заказом?»:
- Кем и когда заказ создан.
- Через какие сервисы прошёл, с какой задержкой.
- Где остановился и почему (ошибка, таймаут, потерянное сообщение).

Сейчас для расследования инцидента нужно вручную искать в логах трёх разных сервисов, сопоставлять по времени. Трейсинг сводит это к одному запросу по order_id.

### Метрики, на которые повлияет внедрение

**Технические:**
1. **MTTR (Mean Time to Resolve)** — среднее время устранения инцидента. Сейчас расследование потерянного заказа занимает много времени и ресурсы разработки. С трейсингом можно будет быстро найти проблему для конкретного кейса.
2. **Процент заказов с полной трассировкой** — доля заказов, для которых есть trace от создания до завершения. Целевое значение: >99%. Позволяет оценить качество инструментирования.
3. **Количество «разрывов» в трейсах** — сколько трейсов обрывается на каком-то шаге (span без продолжения). Показывает, где именно теряются заказы, и в каком количестве.

**Бизнесовые:**
4. **Время прохождения заказа по этапам (end-to-end lead time)** — из данных трейсинга можно извлечь реальное время каждого этапа и найти узкие места.
5. **Количество жалоб на потерянные заказы** — с проактивным мониторингом трейсов команда будет находить проблемы до того, как клиент позвонит.

## 3. Предлагаемое решение

### Выбор технологии

**OpenTelemetry** — как единый стандарт инструментирования. Поддерживает Java (Shop API, CRM API) и .NET (MES API).

**Jaeger** — как бэкенд для хранения и визуализации трейсов. Open-source, хорошо интегрируется с OpenTelemetry, имеет UI для поиска трейсов по атрибутам (в т.ч. по order_id).

## 4. Компромиссы

### Где трейсинг не поможет или будет дорогим

**Расчёт стоимости внутри MES.** MES — купленная программа. Если для логики расчёта мы можем обернуть только вход и выход,то ирейс покажет «расчёт начался → расчёт завершился за 25 мин», но не объяснит, почему именно 25 минут.

## 5. Безопасность

### Защита данных трейсинга

Трейсы содержат внутреннюю информацию о системе: имена сервисов, структуру запросов, идентификаторы заказов и клиентов. Несанкционированный доступ к этим данным — риск.

**Внутренний контур (сотрудники компании):**
- **Доступ к Jaeger UI** — только через VPN или внутреннюю сеть. Jaeger не выставляется в публичный интернет.
- **Аутентификация** — через корпоративную систему. Доступ только для ролей «Разработчик», «DevOps», «Поддержка».
- **Ролевое разграничение** — Jaeger UI в read-only. Право удаления трейсов и изменения конфигурации — только у DevOps.

**Внешний контур:**
- API-ключи и персональные данные клиентов не записываются в трейсы. client_id — обезличенный идентификатор.

**Защита каналов передачи:**
- Защищенный зашифрованный трафик между приложениями

**Хранение:**
- Храним трейсы — 21 день. Не хранить дольше, чем необходимо для диагностики.
