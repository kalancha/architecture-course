# Архитектурное решение по кешированию

## 1. Мотивация

### Проблема

Операторы жалуются на медленную загрузку первой страницы MES — дашборда со списком заказов по статусам. Команда уже попробовала добавить пагинацию и фильтр по статусам, но это не помогло. Помимо этого, клиентов (B2B и B2C) не устраивает скорость обработки заказа в целом.

При анализе мы определили, что основная причина тормозов дашборда — конкуренция за CPU между расчётом стоимости и обслуживанием HTTP-запросов на одном инстансе MES API. Вынесение расчёта в отдельный воркер устранит эту корневую причину. Однако кеширование решает другую, дополнительную проблему: каждый оператор при каждом открытии или обновлении страницы генерирует запрос к БД. Несколько десятков операторов, обновляющих дашборд каждые 10–30 секунд, создают повторяющуюся нагрузку на MES DB — одни и те же данные запрашиваются снова и снова, хотя статусы заказов меняются относительно редко.

### Что предлагается закешировать

1. Список заказов для дашборда MES (фильтр по статусу + пагинация). Это основной источник повторяющихся запросов к БД.

2. Результаты расчёта стоимости. Расчёт занимает от 2 до 30 минут и зависит от 3D-модели. Если один и тот же файл отправляется повторно (например, типовая модель из каталога или повторный заказ от B2B-партнёра), можно вернуть уже посчитанную стоимость без повторного расчёта.

3. Справочные данные: каталог изделий в онлайн-магазине, списки статусов, конфигурации. Эти данные меняются редко и запрашиваются часто.

### Чего мы ожидаем от внедрения

- Дашборд MES загружается быстрее, потому что большинство запросов обслуживаются из кеша, а не из БД.
- Нагрузка на MES DB снижается, что в свою очередь улучшает время отклика для всех остальных операций (запись статусов, поиск заказов).
- Повторные расчёты стоимости для идентичных моделей не выполняются — экономия от 2 до 30 минут CPU-времени на каждый повторный запрос.

## 2. Предлагаемое решение

### Клиентское или серверное кеширование

Для дашборда MES и расчёта стоимости нужно серверное кеширование. Клиентское кеширование (HTTP-кеш в браузере через Cache-Control заголовки) здесь не подходит по нескольким причинам:

- Дашборд показывает данные, которые меняются другими пользователями и системами (оператор А взял заказ — оператор Б должен это увидеть). Если кешировать на клиенте, каждый оператор будет видеть устаревший список, а в контексте компании операторы получают оплату за взятые заказы — несвежие данные приведут к конфликтам.
- B2B-клиенты работают через API, а не через браузер. Клиентского кеша у них может не быть вовсе.
- Управление инвалидацией на клиенте ненадёжно — нет способа принудительно обновить кеш у всех операторов одновременно при изменении статуса.

Для каталога интернет-магазина клиентское кеширование тоже уместно (Cache-Control с max-age для статических данных каталога), но основная проблема сейчас не в магазине, а в MES, поэтому фокусируемся на серверном кеше.

### Выбор технологии

Redis — отдельный инстанс рядом с MES API. Redis работает в памяти, даёт латентность порядка миллисекунды, поддерживает TTL и структуры данных, удобные для хранения списков и хэшей.

### Выбор паттерна кеширования

#### Сравнительный анализ

#### Выбор: Cache-Aside

Для дашборда MES и результатов расчёта стоимости используем Cache-Aside. Причины:

- Данные дашборда читаются часто (десятки операторов обновляют страницу), а пишутся редко (статус заказа меняется несколько раз за его жизненный цикл). Это классический случай для Cache-Aside — кеш заполняется при первом чтении, последующие чтения обслуживаются из кеша.

- В MES DB данные записываются из нескольких источников: CRM через RabbitMQ, операторы через MES UI, B2B-клиенты через API. При Write-Through пришлось бы гарантировать, что каждый из этих путей обновляет кеш, иначе данные рассогласуются. С Cache-Aside достаточно инвалидировать кеш при любой записи — проще и надёжнее.

- Refresh-Ahead требует заранее знать, какие ключи нужно обновлять, и запускать фоновый процесс. Набор ключей зависит от фильтров и страниц пагинации, он динамический — предсказать его сложно, а реализация получается неоправданно громоздкой.

### Структура кеша

| Ключ | Значение | TTL | Инвалидация |
|------|----------|-----|-------------|
| `orders:status:{status}:page:{page}` | JSON со списком заказов для дашборда | 30 сек | По событию: при изменении статуса любого заказа |
| `price:{model_hash}` | Результат расчёта стоимости | 7 дней | Не инвалидируется (для одной и той же модели цена не меняется) |
| `catalog:items:page:{page}` | Страница каталога магазина | 5 мин | По событию: при обновлении каталога |

### Диаграммы последовательности

#### Чтение списка заказов (дашборд MES)

[sequence-read-orders.drawio](sequence-read-orders.drawio)

Описание потока:
1. Оператор открывает дашборд MES → MES frontend отправляет запрос в MES API (GET /orders?status=X&page=Y).
2. MES API формирует ключ кеша `orders:status:X:page:Y` и обращается к Redis.
3. Если в Redis есть данные (cache hit) — возвращаем их сразу, без обращения к БД.
4. Если данных нет (cache miss) — запрашиваем из MES DB, кладём результат в Redis с TTL 30 секунд, возвращаем оператору.

#### Изменение статуса заказа (запись)

[sequence-update-status.drawio](sequence-update-status.drawio)

Описание потока:
1. Оператор нажимает «Взять в работу» → MES frontend отправляет запрос в MES API (POST /orders/{id}/status).
2. MES API записывает новый статус в MES DB.
3. После успешной записи MES API инвалидирует связанные ключи в Redis: удаляет ключи `orders:status:{old_status}:*` и `orders:status:{new_status}:*`.
4. Следующий запрос дашборда с этими статусами получит cache miss и загрузит свежие данные из БД.

### Стратегия инвалидации

#### Сравнительный анализ стратегий

| Стратегия | Суть | Плюсы | Минусы | Подходит ли нам |
|-----------|------|-------|--------|-----------------|
| Временная (TTL) | Кеш устаревает автоматически через заданное время | Простая реализация, нулевая связность | Данные устаревают на весь TTL — оператор может не увидеть свежий заказ | Частично — как страховка, но не как основная стратегия |
| По событию (программная) | При изменении данных приложение явно удаляет или обновляет кеш | Данные в кеше всегда близки к актуальным | Нужно не пропустить ни одного пути записи | Да — основная стратегия |
| По ключу (ручная) | Администратор вручную сбрасывает кеш | Полный контроль | Не масштабируется, требует человека | Нет |

#### Выбор: событийная инвалидация + TTL как страховка

Мы используем комбинацию двух стратегий:

Основная — событийная (программная) инвалидация. При любом изменении статуса заказа (из MES UI, из RabbitMQ, из API) приложение удаляет из Redis ключи, связанные с затронутыми статусами. Это гарантирует, что следующий запрос дашборда получит свежие данные.

Дополнительная — TTL 30 секунд как страховка. Если по какой-то причине событийная инвалидация не сработала (например, ошибка в коде, новый путь записи, который забыли обернуть инвалидацией), данные всё равно обновятся через 30 секунд. Для оператора это приемлемая задержка — разница между «вижу заказ мгновенно» и «вижу через 30 секунд» не критична, если обычно он появляется сразу.

Почему не только TTL: если поставить TTL без событийной инвалидации, то при TTL 30 секунд оператор может не увидеть новый заказ до полуминуты. Если увеличить TTL для снижения нагрузки на БД — задержка вырастет ещё сильнее. Событийная инвалидация позволяет держать TTL коротким и при этом показывать данные почти в реальном времени.

Почему не только событийная инвалидация: в системе есть несколько путей изменения данных (операторы, RabbitMQ от CRM, B2B API). Есть риск, что при доработке кода или добавлении нового эндпоинта разработчик забудет добавить инвалидацию. TTL страхует от таких человеческих ошибок.

#### Особый случай: кеш результатов расчёта стоимости

Для ключа `price:{model_hash}` используется только TTL (7 дней), без событийной инвалидации. Расчёт стоимости детерминированный — для одной и той же 3D-модели результат всегда одинаковый. Если формула расчёта изменится (обновление прайса, изменение стоимости материалов), потребуется ручной сброс всех price-ключей. Это редкое событие, и для него достаточно одной команды в Redis

## 3. Компромиссы и ограничения

Кеширование не устраняет корневую причину тормозов дашборда. Если расчёт стоимости продолжает работать на том же инстансе, что и API, кеш не поможет в моменты высокой загрузки CPU — Redis ответит быстро, но MES API может не успеть обработать HTTP-запрос. Кеширование нужно внедрять вместе с вынесением расчёта в воркер, а не вместо него.

Инвалидация по паттерну (`orders:status:X:*`) удаляет все страницы для данного статуса. При большом количестве страниц это может привести к кратковременному увеличению нагрузки на БД (все страницы перезапросятся). На текущем масштабе это некритично, но при значительном росте стоит перейти на более точечную инвалидацию.

Кеш занимает память. При текущем объёме данных (несколько тысяч заказов) это единицы мегабайт — минимальный инстанс Redis справится. Но если логика кеширования будет расширяться, нужно следить за потреблением памяти через мониторинг.
