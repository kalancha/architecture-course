# Архитектурное решение по логированию

## 1. Мотивация

### Зачем нужно централизованное логирование

Сейчас разбор инцидентов строится со слов клиента: клиент описывает проблему менеджеру, менеджер идёт к разработчику, разработчик подключается к EC2-инстансу, ищет в локальных логах по времени и ключевым словам. Если заказ прошёл через несколько сервисов (Shop API → RabbitMQ → MES API → CRM API), нужно искать в логах каждого инстанса отдельно и вручную сопоставлять по времени.

Трейсинг покажет путь заказа через сервисы, но не покажет контекст — что именно происходило внутри сервиса: какие данные пришли в запросе, какая валидация не прошла, почему бизнес-логика вернула ошибку, какой SQL-запрос выполнялся. Логи — это детальный бортовой журнал для сервиса.

Централизованное логирование объединяет логи всех сервисов в одном месте с возможностью поиска, фильтрации и корреляции.

### Метрики, на которые повлияет внедрение

**Технические:**
1. **MTTD (Mean Time to Detect)** — среднее время обнаружения проблемы. Сильно уменьшится.
2. **MTTR (Mean Time to Resolve)** — среднее время устранения инцидента. Сильно уменьшится.

**Бизнесовые:**
4. **Нагрузка на разработчиков от поддержки** — сейчас каждое обращение клиента требует подключения разработчика. С логами поддержка справляется самостоятельно для типовых случаев. Высвобождается время разработчиков на новые фичи.
5. **Обнаружение аномалий и атак** — всплеск создания заказов, массовые 4xx/5xx, необычные паттерны — система алертит до того, как проблема масштабируется.

## 2. Анализ: какие логи собирать и откуда

### Системы, из которых требуется сбор логов

Логи необходимо собирать из всех компонентов системы:

- **Shop API** (Java/Spring Boot) — точка входа B2C
- **CRM API** (Java/Spring Boot) — работа с заказами, подтверждение производства
- **MES API** (C#/.NET) — точка входа B2B, расчёт стоимости, управление производством
- **RabbitMQ** — логи брокера сообщений (ошибки, подключения/отключения consumers)
- **Shop DB, MES DB** (PostgreSQL) — ошибки, события репликации

### Список логов уровня INFO

#### Shop API

| Событие | Поля | Зачем |
|---------|------|-------|
| Создание заказа | timestamp, order_id, customer_id, order_source, trace_id | Точка входа заказа в систему |
| Загрузка 3D-файла | timestamp, order_id, customer_id, file_name, file_size_kb, trace_id | Диагностика проблем с загрузкой, выявление аномально больших файлов |
| Файл сохранён в S3 | timestamp, order_id, s3_key, duration_ms, trace_id | Подтверждение успешной загрузки, измерение latency |
| Отправка заказа | timestamp, order_id, customer_id, trace_id | События отправки заказа в работу - как точка входа в процесс работы над заказом |
| Сообщение отправлено в RabbitMQ | timestamp, order_id, queue_name, message_id, trace_id | Подтверждение передачи заказа в очередь |
| Изменение статуса заказа | timestamp, order_id, customer_id, old_status, new_status, trace_id | Полная история переходов статуса |

#### CRM API

| Событие | Поля | Зачем |
|---------|------|-------|
| Получение сообщения из RabbitMQ | timestamp, order_id, queue_name, message_id, trace_id | Подтверждение, что CRM получил заказ/статус |
| Подтверждение производства | timestamp, order_id, seller_id, trace_id | Кто и когда подтвердил — для аудита |
| Закрытие заказа | timestamp, order_id, close_reason, trace_id | Завершение жизненного цикла заказа |
| Отправка сообщения в RabbitMQ | timestamp, order_id, queue_name, message_id, trace_id | Подтверждение передачи команды в MES |

#### MES API

| Событие | Поля | Зачем |
|---------|------|-------|
| Получение заказа через API (B2B) | timestamp, order_id, client_id, order_source=api, trace_id | Точка входа B2B-заказа |
| Получение сообщения из RabbitMQ | timestamp, order_id, queue_name, message_id, trace_id | Подтверждение получения |
| Начало расчёта стоимости | timestamp, order_id, model_hash, polygon_count, trace_id | Старт CPU-intensive операции, данные о сложности модели |
| Завершение расчёта стоимости | timestamp, order_id, calculated_price, duration_ms, trace_id | Результат расчёта и его длительность |
| Оператор взял заказ | timestamp, order_id, operator_id, trace_id | Кто взял заказ — важно для отслеживания работы над заказом |
| Заказ завершён | timestamp, order_id, operator_id, duration_days, trace_id | Время производства |
| Упаковка | timestamp, order_id, operator_id, trace_id | Этап перед отправкой |
| Отправка | timestamp, order_id, tracking_number, trace_id | Связь с логистикой |
| Отправка сообщения в RabbitMQ | timestamp, order_id, queue_name, message_id, trace_id | Подтверждение передачи |
| Загрузка дашборда | timestamp, operator_id, filter_status, page, response_time_ms, trace_id | Диагностика проблемы с дашбордом |

### Уровни логирования

| Уровень | Когда использовать | Примеры |
|---------|-------------------|---------|
| **INFO** | Штатные бизнес-события, переходы статусов, успешные операции | Все события из таблиц выше |
| **WARN** | Нештатная, но обработанная ситуация. Система справилась, но стоит обратить внимание | Retry отправки в RabbitMQ, медленный ответ от БД, файл превышает рекомендуемый размер, приближение к лимиту connections в БД |
| **ERROR** | Ошибка, которую сервис не смог обработать. Требуется внимание | HTTP 500, невозможность подключения к БД, сообщение попало в DLQ, ошибка записи в S3, ошибка расчёта стоимости, таймаут RabbitMQ |
| **DEBUG** | Детальная информация для разработки. Включается временно для диагностики конкретной проблемы | Содержимое HTTP-запроса/ответа, SQL-запросы, тело сообщения RabbitMQ. |

## 3. Приоритизация: порядок внедрения логирования и трейсинга

Команда не может внедрить логирование и трейсинг во все системы одновременно. Порядок внедрения:

### Фаза 1: MES API

**Почему MES первый:**
- MES API — точка, где сходятся обе главные проблемы: потеря заказов (B2B API + расчёт стоимости) и тормозящий дашборд.
- MES обслуживает три категории пользователей (операторы, B2B-партнёры, внутренние процессы) — максимальное покрытие при одной интеграции.
- Именно в MES выполняется расчёт стоимости (2–30 мин) — основной чёрный ящик системы. Логирование начала/конца расчёта с polygon_count и duration_ms даст базу для оптимизации.
- C# стек — один разработчик в команде. Лучше начать с него, пока он свободен, а не ждать очереди.

### Фаза 2: Shop API + CRM API

**Почему вместе:** оба на Java/Spring Boot, один разработчик настраивает инфраструктуру логирования (Logback → Elasticsearch), второй добавляет бизнес-логи. Конфигурация переиспользуется между двумя сервисами.

**Почему после MES:** MES — более критичная точка отказа. Shop API и CRM API менее проблемны с точки зрения текущих жалоб.

### Фаза 3: RabbitMQ + БД

Логи инфраструктурных компонентов подключаются последними — у них меньше бизнес-контекста, но они полезны для глубокой диагностики

## 4. Предлагаемое решение

### Выбор технологии

**ELK Stack:**
- **Elasticsearch** — хранение и полнотекстовый поиск по логам.
- **Kibana** — UI для поиска, фильтрации и визуализации логов. Поддержка сможет искать по order_id без привлечения разработчиков.
- **Filebeat** — легковесный агент на каждом EC2-инстансе, отправляет логи в Elasticsearch.

**Почему ELK:** стандарт индустрии для централизованного логирования, open-source, хорошая интеграция с Java и .NET стеками. Kibana даёт удобный UI для нетехнических пользователей.

### Формат логов

Все сервисы пишут логи в структурированном JSON-формате:

```json
{
  "timestamp": "2026-04-21T12:34:56.789Z",
  "level": "INFO",
  "service": "mes-api",
  "event": "price_calculation_completed",
  "order_id": "ORD-12345",
  "trace_id": "abc123def456",
  "client_id": "B2B-42",
  "calculated_price": 15400.00,
  "duration_ms": 142000,
  "message": "Price calculation completed for order ORD-12345"
}
```

Структурированные логи позволяют фильтровать и агрегировать в Kibana по любому полю. trace_id связывает лог с трейсом в Jaeger.

### Архитектура

**C4-диаграмма с компонентами логирования:** [C4-logging.drawio](C4-logging.drawio)

Новые компоненты (выделены оранжевым на диаграмме):
- **Filebeat** — агент рядом с каждым API-сервисом, собирает JSON-логи из файлов и отправляет в Elasticsearch.
- **Elasticsearch** — централизованное хранение логов, полнотекстовый поиск, алертинг.
- **Kibana** — UI для поиска, фильтрации, визуализации логов и настройки алертов.

## 5. Политика безопасности

### Чувствительные данные

Логи не должны содержать:
- **Пароли, токены, API-ключи** — маскируются на уровне приложения (redaction).
- **Персональные данные клиентов** (ФИО, email, телефон, адрес) — используются обезличенные идентификаторы (customer_id, client_id).
- **Полные номера банковских карт** — недопустимо по PCI DSS.
- **Тело HTTP-запроса/ответа** — только на уровне DEBUG, который по умолчанию выключен в prod.


### Контроль доступа

| Роль | Доступ в Kibana | Что видит |
|------|----------------|-----------|
| **DevOps** | Полный | Все индексы, настройка дашбордов и алертов, управление |
| **Разработчик** | Чтение | Все индексы, поиск по любым полям |
| **Поддержка** | Чтение (ограниченное) | Только бизнес-логи (INFO-уровень), поиск по order_id, customer_id. Без доступа к DEBUG и системным логам |
| **Продакт-менеджер** | Дашборды | Только настроенные дашборды (воронка заказов, статистика ошибок) |

Доступ к Kibana — только через VPN/внутреннюю сеть. Аутентификация через корпоративную систему. Kibana не выставляется в публичный интернет.

## 6. Политика хранения

### Индексация

Отдельный индекс на каждый сервис + период:

| Индекс (паттерн) | Источник | Обоснование |
|-------------------|----------|-------------|
| `shop-api-YYYY.MM.DD` | Shop API | Изоляция: проблемы с индексом одного сервиса не влияют на другие |
| `crm-api-YYYY.MM.DD` | CRM API | |
| `mes-api-YYYY.MM.DD` | MES API | Самый объёмный — расчёты, дашборд, B2B |
| `rabbitmq-YYYY.MM.DD` | RabbitMQ | Инфраструктурные логи — отдельно |
| `postgres-YYYY.MM.DD` | Shop DB + MES DB | Slow queries + ошибки |

Ежедневная ротация (daily index) позволяет эффективно удалять старые данные целыми индексами.

### Сроки хранения

| Тип логов | Срок хранения | Обоснование |
|-----------|--------------|-------------|
| Бизнес-логи API (INFO) | 30 дней | Покрывает типичный цикл жизни заказа (производство + доставка ~3–4 недели) |
| Ошибки (ERROR, WARN) | 90 дней | Для анализа трендов и регрессий между релизами |
| DEBUG | 3 дня | Включается временно, большой объём, хранить долго бессмысленно |
| Инфраструктурные | 14 дней | Нужны только для оперативной диагностики |

### Оценка объёма

При текущей нагрузке (~100 заказов/мес, несколько десятков операторов):
- Оценка: ~1–5 ГБ/день на все сервисы (без DEBUG).
- С retention 30 дней: ~50–150 ГБ в Elasticsearch.

Настроить ILM в Elasticsearch для автоматического удаления индексов по истечении срока хранения.

## 7. От сбора к анализу: алертинг и обнаружение аномалий

### Алертинг на основе логов

Kibana поддерживает алертинг через встроенные rules. Настраиваем:

| Алерт | Условие | Серьёзность | Канал |
|-------|---------|-------------|-------|
| Всплеск ошибок | Количество ERROR > 10 за 5 минут в любом сервисе | Critical | Telegram |
| Сообщение в DLQ | ERROR с event=`message_dead_lettered` | Critical | Telegram + email |
| Медленные SQL-запросы | Более 5 записей slow query за 10 минут | Warning | Telegram |
| Ошибки подключения к БД | ERROR с event=`db_connection_failed` | Critical | Telegram |

### Обнаружение аномалий

**Зачем:** выявление DDoS-атак, злоупотреблений через API, нехарактерного поведения.

Настроить в Kibana:

1. **Аномальный рост создания заказов.** Базовая линия: N заказов в час (скользящее среднее за неделю). Алерт если текущее значение > 5x от базовой линии за 10 минут.
   - Выявляет: DDoS, злоупотребление API, ботов.

2. **Аномальный рост 4xx ошибок на Shop API.** Базовая линия: M ошибок 4xx в час. Алерт если > 10x.
   - Выявляет: сканирование уязвимостей, brute force, некорректные интеграции B2B.

3. **Аномальный рост 5xx ошибок.** Любой сервис: > 5% запросов с 5xx за 5 минут.
   - Выявляет: деградацию после деплоя, каскадный отказ, проблемы с БД.

4. **Аномальный объём загрузки файлов.** Резкий рост количества или суммарного объёма загружаемых 3D-файлов.
   - Выявляет: злоупотребление хранилищем, атаку на S3.

5. **Необычная активность одного client_id (B2B).** Один клиент генерирует > 80% трафика за час.
   - Выявляет: некорректно настроенную интеграцию, злоупотребление.
