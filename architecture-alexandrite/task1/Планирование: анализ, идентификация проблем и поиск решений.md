# Планирование: анализ, идентификация проблем и поиск решений

## 1. Идентификация существующих и потенциальных проблем

### 1.1. Существующие проблемы

#### P1. Потеря заказов

**Симптомы:**
- Клиенты онлайн-магазина жалуются, что заказы не выполняются месяцами вместо обещанных трёх недель.
- Представители компаний-пользователей API ежедневно сообщают о потерянных заказах.
- Компания потеряла несколько крупных контрактов.

**Анализ корневых причин:**
Производственные мощности позволяют справиться с нагрузкой (с запасом до 2x), значит проблема не в производстве, а в цепочке обработки заказов в софте.

- **Единственный инстанс каждого сервиса.** Если MES API, CRM API, Shop API или RabbitMQ падает — заказы теряются. При одном EC2-инстансе на сервис нет никакого резервирования. Перезапуск означает простой, а в это время входящие запросы/сообщения отбрасываются.
- **Ненадёжная обработка сообщений в RabbitMQ.** CRM и MES взаимодействуют через очередь. Если consumer не подтверждает (ack) сообщение перед падением, сообщение может быть потеряно
- **Расчёт стоимости — длительная операция (2–30 мин).** Если MES-инстанс перезагружается во время расчёта, состояние вычисления теряется. Нет механизма возобновления (retry/resume). При росте заказов расчёты скапливаются, и один инстанс не справляется — образуется очередь, которая выглядит как потерянные заказы.
- **Отсутствие мониторинга и алертинга.** Команда узнаёт о проблемах от пользователей, а не от системы. Нет способа отследить, где заказ «застрял» в цепочке статусов.

#### P2. Медленная загрузка дашборда MES

**Симптомы:**
- Операторы жалуются на долгую загрузку первой страницы MES (список заказов по статусам).
- Пагинация и фильтр по статусам не помогли.

**Анализ корневых причин:**
- **Единственный инстанс MES DB работает и на чтение, и на запись.** Тяжёлые операции записи (обновление статусов, расчёт стоимости) блокируют или замедляют операции чтения.
- **Отсутствие кеширования.** Каждый запрос дашборда обращается напрямую в БД. При нескольких десятках операторов, одновременно обновляющих страницу, нагрузка на БД возрастает.
- **Рост объёма данных.** Компания получает линейно растущее число заказов (+100/мес). Без архивации исторических данных таблицы разрастаются, запросы замедляются.
- **Конкуренция за ресурсы CPU/IO на одном инстансе MES API.** Расчёт стоимости выполняется на том же инстансе, что обслуживает HTTP-запросы дашборда. Тяжёлые вычисления отъедают ресурсы у API.

#### P3. Нарастающие задержки релизов

**Симптомы:**
- Задержки релизов раньше не превышали месяца, теперь растут.
- Один QA-инженер прогоняет E2E-тесты вручную.

**Анализ корневых причин:**
- **Ручное тестирование не масштабируется.** С ростом функциональности количество E2E-сценариев растёт, а пропускная способность одного QA-инженера — нет.
- **Критические баги блокируют деплой всего.** Монолитный цикл релиза: баг в одном приложении блокирует выпуск всех изменений.
- **Отсутствие автоматизированных тестов в CI/CD.** Только юнит-тесты запускаются автоматически; E2E, интеграционные и нагрузочные тесты — нет.

### 1.2. Потенциальные проблемы

| # | Проблема | Риск |
|---|----------|------|
| P4 | **Нет горизонтального масштабирования** — по одному инстансу на сервис, нет load balancer | При двукратном росте нагрузки система упрётся в потолок одного инстанса |
| P5 | **Нет rate limiting / throttling на API для B2B** | Один клиент может положить MES API для всех, включая операторов |
| P6 | **Нет автомасштабирования (autoscaling)** | При пиковых нагрузках (распродажи, праздники) система упадет |
| P7 | **БД без read-реплик** | С ростом числа читающих клиентов (операторы, дашборды, API) база станет узким местом |
| P8 | **Нет circuit breaker между сервисами** | Каскадные отказы: если MES DB тормозит, MES API зависает, RabbitMQ забивается, CRM перестаёт работать |
| P9 | **Нет health checks и readiness probes** | Невозможно автоматически определить, что сервис «жив», но не функционален |
| P10 | **Нет трассировки запросов (distributed tracing)** | Невозможно отследить путь заказа через 3 системы и очередь |
| P11 | **Нет стратегии архивации данных** | Рост таблиц будет продолжать замедлять запросы |

## 2. Инициативы для устранения проблем

### I1. Внедрение системы мониторинга, алертинга и трассировки

**Решает:** P1 (частично), P9, P10

**Описание:**
- Развернуть стек мониторинга: Prometheus + Grafana для метрик, ELK для логов.
- Внедрить tracing (Jaeger) для отслеживания пути заказа через все системы.
- Настроить алерты на: рост очереди в RabbitMQ, время ответа API > пороговых значений, 5xx-ошибки, задержку обработки заказов.
- Добавить health checks и readiness probes для всех сервисов.
- Создать дашборд «воронки заказов» — визуализацию количества заказов в каждом статусе и среднего времени перехода.

**Оценка:** 3–4 недели (1 DevOps + 1 backend-разработчик). Минимальные изменения в коде приложений — в основном инфраструктура и инструментирование.

---

### I2. Внедрение кеширования для MES-дашборда и горячих данных

**Решает:** P2, P7 (частично)

**Описание:**
- Развернуть Redis как кеш-слой.
- Кешировать результаты запросов дашборда MES (список заказов по статусам с пагинацией). TTL: 5–15 секунд (операторам достаточно near-real-time данных).
- Инвалидация кеша при изменении статуса заказа (event-driven: при записи в БД обновляем/сбрасываем кеш).
- Кешировать справочные данные: каталог изделий, список операторов, справочники статусов.
- Рассмотреть кеширование результатов расчёта стоимости для типовых 3D-моделей (если одну и ту же модель заказывают повторно — не считать заново).

**Стратегия кеширования:**
- **Дашборд MES:** Cache-aside pattern. Ключ = `orders:status:{status}:page:{page}`. Инвалидация при любом изменении статуса заказа.
- **Каталог товаров магазина:** Cache-aside с TTL 5 минут. Инвалидация при обновлении каталога.
- **Расчёт стоимости:** Кешировать по хэшу 3D-модели. TTL длительный (дни), т.к. для одной и той же модели цена не меняется.

**Оценка:** 2–3 недели (1 backend C# для MES, 1 DevOps для инфраструктуры Redis).

---

### I3. Обеспечение надёжности обработки заказов (reliability pipeline)

**Решает:** P1, P8

**Описание:**
- **RabbitMQ:** настроить durable queues, persistent messages, manual acknowledgement, Dead Letter Queue (DLQ) для необработанных сообщений. Настроить retry-политику с exponential backoff.
- **Circuit breaker** между сервисами (Polly для C#, Resilience4j для Java): при недоступности downstream-сервиса не копить запросы.
- **Мониторинг DLQ:** алерт при появлении сообщений в DLQ.

**Оценка:** 2–3 недели (1 backend Java + 1 backend C#).

---

### I4. Вынесение расчёта стоимости в отдельный воркер-процесс

**Решает:** P1 (частично), P2 (частично), P4 (частично)

**Описание:**
- Расчёт стоимости — CPU-intensive задача (2–30 мин). Сейчас она выполняется внутри MES API, что забирает ресурсы у обслуживания HTTP-запросов (дашборд, API).
- Вынести расчёт в отдельный worker-сервис (background job), который:
  - Читает задания на расчёт из очереди (RabbitMQ или отдельная очередь).
  - Выполняет расчёт изолированно.
  - Записывает результат в MES DB и отправляет событие `PRICE_CALCULATED`.
  - Поддерживает retry при падении (задание возвращается в очередь).
- Worker можно масштабировать горизонтально независимо от MES API: при росте заказов просто добавляем инстансы воркеров.

**Оценка:** 3–4 недели (1 backend C# + DevOps). Это рефакторинг MES API.

---

### I5. Горизонтальное масштабирование и балансировка нагрузки

**Решает:** P4, P6

**Описание:**
- Развернуть Application Load Balancer перед каждым API-сервисом.
- Минимум 2 инстанса каждого сервиса (MES API, Shop API, CRM API) для отказоустойчивости.
- Настроить auto-scaling groups с политиками по CPU/memory.
- RabbitMQ: перейти на кластер из 3 нод (quorum queues) для отказоустойчивости.
- Подключить read-реплики для MES DB и Shop DB (направлять чтение дашборда на реплику).

**Оценка:** 3–4 недели (DevOps + 1 backend). Требует обеспечения stateless-архитектуры сервисов (вынести сессии и состояние во внешние хранилища).

---

### I6. Rate limiting и защита API для B2B-клиентов

**Решает:** P5

**Описание:**
- Внедрить API Gateway (или rate limiting на уровне ALB / nginx) перед MES API.
- Лимиты по API-ключу: X запросов/минуту, Y запросов/час.
- Отдельные квоты для разных тарифов B2B-клиентов.
- Возврат HTTP 429 (Too Many Requests) с заголовком Retry-After.
- Приоритизация: внутренние запросы (операторы, CRM) имеют более высокий приоритет, чем внешние API-клиенты.

**Оценка:** 1–2 недели (DevOps + 1 backend).

---

### I7. Автоматизация тестирования и ускорение CI/CD

**Решает:** P3

**Описание:**
- Внедрить автоматические E2E-тесты (Playwright / Cypress) для критических сценариев: оформление заказа, смена статусов, работа дашборда.
- Добавить интеграционные тесты для взаимодействия через RabbitMQ (testcontainers).
- Включить E2E-тесты в CI-пайплайн для release-окружения.
- Разделить монолитный релизный цикл: позволить деплоить каждый сервис независимо (independent deployability).
- Нагрузочное тестирование (k6 / Gatling) перед релизом на staging.

**Оценка:** 4–6 недель (QA + 1 backend). QA-инженер хочет заниматься автоматизацией — это его возможность.

---

### I8. Оптимизация базы данных и архивация

**Решает:** P2 (частично), P11

**Описание:**
- Провести анализ медленных запросов (pg_stat_statements) в MES DB и Shop DB.
- Добавить составные индексы для запросов дашборда (status + created_at).
- Реализовать архивацию завершённых заказов (статус CLOSED старше N месяцев) в отдельную таблицу/партицию.
- Рассмотреть партиционирование таблицы заказов по дате создания.

**Оценка:** 1–2 недели (1 backend + DBA-консультация).

## 3. Приоритизация инициатив

### Матрица приоритизации

| Приоритет | Инициатива | Impact | Effort | Обоснование |
|-----------|-----------|--------|--------|-------------|
| **1 (критический)** | I1. Мониторинг и алертинг (+ DLQ ) | Высокий | Средний | Это фундамент для всех остальных инициатив. В рамках I1 настраиваем DLQ и алертинг на очереди — дешёвое дополнение, которое сразу ловит потерянные заказы. |
| **2 (критический)** | I4. Вынесение расчёта в воркер | Высокий | Средний | Устраняет корневую причину тормозов дашборда MES (CPU-конкуренция) и ненадёжности расчёта стоимости. Одна инициатива решает две крупные проблемы. |
| **3 (критический)** | I7. Автоматизация тестирования + разделение деплоя | Высокий | Высокий | Time-to-market — ключевое конкурентное преимущество для растущего бизнеса. Задержки релизов — самоусиливающаяся проблема: с каждым спринтом становится хуже. Разделение деплоя сервисов даёт структурное ускорение всей команды. |
| **4 (высокий)** | I3. Надёжность обработки заказов | Высокий | Средний | Важно, но целесообразно делать итеративно по результатам мониторинга (I1), а не везде сразу. Базовая надёжность RabbitMQ (DLQ) уже входит в I1. |
| **5 (высокий)** | I5. Горизонтальное масштабирование | Высокий | Высокий | Необходимо для устойчивого роста, но требует существенной работы с инфраструктурой. Без I1 (мониторинг) сложно настраивать autoscaling. В любом случае, учитывая дальнейший рост и работу над надежностью это рано или поздно придется сделать |
| **6 (средний)** | I8. Оптимизация БД | Средний | Низкий | Индексы и архивация дадут эффект для дашборда за 1–2 недели. Выполняется параллельно с основными инициативами. |
| **7 (средний)** | I2. Кеширование | Средний | Низкий | Полезно, но при текущем масштабе нагрузки (+100 заказов/мес) не критично. PostgreSQL справляется с таким объёмом при нормальных индексах. |
| **8 (средний)** | I6. Rate limiting API | Средний | Низкий | Защищает систему от перегрузки B2B-клиентами. Быстро реализуется. |

### Ход рассуждений

Приоритизация строится на четырёх принципах:

1. **Сначала видимость, потом действия.** Без мониторинга мы не знаем масштаб проблемы и не можем измерить эффект от изменений. I1 — фундамент. При этом базовую надёжность RabbitMQ (DLQ + алертинг) дешевле всего внедрить прямо в рамках настройки мониторинга.
2. **Устранять корневые причины, а не симптомы.** Тормоза дашборда MES — это не проблема нагрузки на БД (сервис не highload), а конкуренция за CPU с расчётом стоимости на одном инстансе. Вынесение воркера (I4) устраняет причину.
3. **Time-to-market критичен для растущего бизнеса.** Компания осваивает новые рынки, открыла API, каждый месяц появляются новые требования. Если релизный цикл продолжит замедляться, бизнес не сможет конкурировать. Автоматизация CI/CD и разделение деплоя (I7) — структурное изменение, ускоряющее всю команду.
4. **Надёжность — итеративно, по данным мониторинга.** Полный reliability overhaul избыточен для текущего масштаба. Разумнее внедрить мониторинг, увидеть реальные точки отказа и улучшать прицельно.

## 4. Целевая архитектура через полгода

Через 6 месяцев архитектура «Александрита» должна выглядеть так:

### Ключевые изменения

**Наблюдаемость (Observability):**
- Полноценный стек мониторинга: Prometheus + Grafana для метрик, ELK для логов, Jaeger для трассировки.
- Дашборд «здоровья системы» с SLI/SLO: время обработки заказа, процент успешных переходов статусов, latency API.
- Алертинг в Telegram/Slack при нарушении SLO.

**Расчёт стоимости:**
- Отдельный пул воркеров для расчёта, изолированный от MES API.
- MES API освобождён от CPU-intensive нагрузки — дашборд работает быстро.
- Retry при падении воркера — задание не теряется.

**Надёжность обработки заказов (итеративно):**
- RabbitMQ: durable queues, persistent messages, manual ack, DLQ с алертингом
- Circuit breakers и outbox pattern — точечно, по результатам мониторинга, в местах реальных отказов.

**Кеширование (точечно):**
- Redis для кеширования результатов расчёта стоимости по хэшу 3D-модели.
- Кеш справочных данных (каталог, статусы) с разумным TTL.

**Масштабируемость:**
- MES API и Shop API: минимум 2 инстанса за ALB, auto-scaling group.
- Расчёт стоимости: отдельный пул воркеров, масштабируемый по длине очереди задач.
- Read-реплики для MES DB (дашборд операторов читает с реплики).
- RabbitMQ-кластер из 3 нод (quorum queues).

**Защита API:**
- Rate limiting для B2B API (по API-ключу, с приоритизацией внутренних запросов).
- API Gateway перед MES API.

**CI/CD и качество:**
- Автоматические E2E-тесты для критических сценариев в CI-пайплайне.
- Нагрузочное тестирование на staging перед релизом.
- Независимый деплой сервисов (Shop, CRM, MES деплоятся по отдельности).

### Необходимые кадровые изменения

- **+1 backend-разработчик (C# или Java)**: увеличение пропускной способности команды для параллельной работы над инициативами.
- **QA-инженер переходит на автоматизацию**: это уже обосновано его желанием и потребностью бизнеса.
- **+1 SRE/DevOps (по возможности)**: текущий DevOps будет перегружен задачами по мониторингу, масштабированию и инфраструктуре.

## 5. Топ-3 инициативы на ближайшие полгода

Если из всего списка можно выполнить только три инициативы, я выбираю:

### 1. I1 — Мониторинг, алертинг и трассировка (+ базовая надёжность RabbitMQ)

**Почему:** Это фундамент. Сейчас команда узнаёт о проблемах от разгневанных клиентов — реактивно, а не проактивно. Без наблюдаемости любые другие изменения — это стрельба вслепую. Мониторинг позволит:
- Немедленно обнаруживать потерянные заказы (алерт: заказ в статусе X дольше N часов).
- Видеть деградацию производительности до того, как она станет критической.
- Измерять эффект от внедрения других инициатив.
- Понимать реальный профиль нагрузки для планирования масштабирования.

В рамках этой же инициативы настраиваем DLQ в RabbitMQ и алертинг на появление сообщений в DLQ. Это конфигурационная работа на несколько дней, но она сразу поможет реагировать на заказы с ошибками

### 2. I4 — Вынесение расчёта стоимости в отдельный воркер

**Почему:** Эта инициатива устраняет корневую причину сразу двух крупных проблем:
- **Тормозящий дашборд MES.** Расчёт стоимости — CPU-intensive задача на 2–30 минут — выполняется на том же инстансе, что обслуживает HTTP-запросы операторов. Убираем расчёт с инстанса API — дашборд освобождается. Это не проблема нагрузки (сервис не highload), это проблема конкуренции за ресурсы.
- **Ненадёжность расчёта.** Сейчас если MES падает во время расчёта — вычисление теряется, заказ «зависает». С воркером задание живёт в очереди и при падении возвращается на повторную обработку.
- **Масштабируемость.** Воркеры масштабируются горизонтально: при росте заказов просто добавляем инстансы, не трогая MES API.

### 3. I7 — Автоматизация тестирования и разделение деплоя

**Почему:** Компания активно растёт — осваивает новые рынки, открыла API, появляются новые требования. В этой фазе скорость доставки фич — конкурентное преимущество. А сейчас time-to-market деградирует:
- Один QA вручную тестирует три приложения. С каждым спринтом сценариев больше — прогон длиннее.
- Баг в одном сервисе блокирует деплой всех остальных.
- Это самоусиливающаяся проблема: чем дольше не решаем, тем хуже становится.

Инициатива состоит из двух частей:
- **Необходимый минимум (2–3 недели):** разделить деплой на независимые пайплайны для Shop, CRM и MES + автоматизировать smoke-тесты для критических сценариев. QA-инженер уже хочет заниматься автоматизацией — даём ему эту возможность.
- **Полная автоматизация (несрочно):** полный E2E-сьют, интеграционные тесты через testcontainers, нагрузочное тестирование.
