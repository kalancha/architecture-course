# Выбор и настройка мониторинга в системе

## 1. Мотивация

### Почему мониторинг нужен прямо сейчас

Сегодня «Александрит» узнаёт о проблемах от клиентов: звонки менеджерам по продажам, жалобы от B2B-партнёров, потерянные контракты. Команда тушит пожары уже после того, как бизнес пострадал.

С момента открытия API Яндекс Метрика перестала давать полную картину. Метрика видит только браузерный трафик онлайн-магазина. Она не видит:
- Заказы, поступающие через API от B2B-партнёров .
- Внутренние процессы: расчёт стоимости в MES, обработку сообщений в RabbitMQ, работу CRM.
- Состояние инфраструктуры: загрузку БД, очередей, CPU сервисов.

Без этих данных невозможно ответить на базовые вопросы: «Сколько заказов застряло?», «Почему дашборд MES тормозит?», «Сколько сообщений потерялось за последнюю неделю?».

### Что мониторинг даст бизнесу

| Проблема бизнеса | Как мониторинг помогает | Измеримый результат |
|---|---|---|
| Потеря заказов → потеря контрактов | Алерт при появлении сообщений в DLQ или при «зависании» заказа в статусе. Команда может отреагировать сразу | Сокращение времени обнаружения проблемы с дней/недель до минут |
| Жалобы операторов на тормозящий дашборд | Метрики latency и CPU показывают, когда и почему дашборд замедляется | Данные для диагностики зависаний, меньше затрат на поиск проблем |
| Жалобы B2B-клиентов на потерянные заказы | Трассировка пути заказа через все системы — видно, где именно он застрял | Быстрая диагностика: минуты вместо часов на разбор инцидента |
| Неясно, справляется ли система с ростом | Метрики утилизации ресурсов и тренды нагрузки | Планирование мощностей на основе данных |

### Обоснование для бизнеса: цена отсутствия мониторинга

Компания уже потеряла несколько крупных B2B-контрактов. Каждый потерянный контракт — это:
- Прямой упущенный доход (стоимость контракта).
- Репутационный ущерб (B2B-партнёры уходят к конкурентам и рассказывают об этом другим).
- Затраты на поддержку (менеджеры тратят время на разбор жалоб вместо продаж).

Стоимость внедрения мониторинга — 3–4 недели работы DevOps и одного backend-разработчика. Стоимость одного потерянного B2B-контракта, вероятнее всего, превышает эти затраты. Мониторинг — необходимая для роста бизнеса функциональность, которая начнет приносить ( экономить ) деньги сразу после внедрения

## 2. Выбор подхода к мониторингу

Для разных частей системы оптимальны разные подходы:

### API-сервисы (Shop API, CRM API, MES API) → RED

**RED** — подход, оптимизированный для request-driven сервисов.

**Почему RED, а не Четыре золотых сигнала»:** RED проще в реализации и покрывает ключевые потребности для наших API. У нас три разных API на двух стеках (Java и C#), и чем проще метрическая модель — тем быстрее внедрение. Четвёртый сигнал (Saturation) для API мы получим из USE-метрик инфраструктуры.

### Инфраструктурные ресурсы (БД, RabbitMQ, EC2, S3) → USE

**USE** — подход, оптимизированный для ресурсов.

**Почему USE:** позволяет заранее увидеть, что ресурс приближается к лимиту, до того как это скажется на пользователях. Критично при одном инстансе на сервис — запаса нет.

### Бизнес-процессы (воронка заказов) → кастомные метрики

Ни RED, ни USE не покрывают бизнес-логику. Нужны отдельные метрики, отслеживающие путь заказа через систему — «воронка заказов». Это уникальные для «Александрита» метрики, не входящие в стандартные подходы.

## 3. Метрики

### 3.1. RED-метрики для API-сервисов

#### RPS for Shop API, CRM API

**Зачем:** базовая метрика трафика. Позволяет видеть текущую нагрузку на онлайн-магазин, обнаруживать аномалии (резкий рост, резкое падение), планировать мощности.

**Ярлыки (labels):**
- `method` — HTTP-метод
- `endpoint` — путь запроса
- `status_code_group` — группа статус-кода (2xx, 4xx, 5xx).

#### RPS for MES API

**Ярлыки:**
- `method`, `endpoint`, `status_code_group` — как для других API.
- `client_type` — тип клиента (`operator`, `b2b_api`, `internal`). Ключевой ярлык: позволяет увидеть нагрузку от разных источников.

#### Response time - Shop API, CRM API, MES API

**Зачем:** следить за скоростью API

**Ярлыки:**
- `method`, `endpoint`, `response_time`,  — для локализации медленных эндпоинтов.

### 3.2. USE-метрики для инфраструктуры

#### CPU % for MES API, SHOP API, CRM API

**Зачем:** видеть потребление CPU, можно сразу делать выводы о причинах проблем на других графиках.

#### Memory Shop API / CRM API / MES API / Shop DB instance / MES DB instance

**Зачем:** следить за утечками памяти, видеть что сервисам хватает памяти

#### Number of connections for Shop DB instance / MES DB instance

**Зачем:** метрика насыщения. PostgreSQL имеет лимит `max_connections`. Приближение к лимиту → новые запросы будут отклоняться → каскадный отказ API.

**Ярлыки:**
- `state` — состояние соединения.

#### Size of Shop DB instance / MES DB instance / S3

**Зачем:** Следить за количеством оставшегося места в БД

### 3.3. Метрики RabbitMQ

#### Number of events in DLQ

**Зачем:** критическая метрика. Каждое сообщение в DLQ — это потенциально потерянный заказ или потерянная смена статуса.

**Ярлыки:**
- `source_queue` — из какой очереди пришло «мёртвое» сообщение (очередь MES→CRM, CRM→MES и т.д.). Позволяет понять, какая интеграция ломается.
- `reason` — причина попадания в DLQ (`rejected`, `expired`, `max_retries_exceeded`).

#### Number of messages in flight in RabbitMQ

**Зачем:** количество сообщений в очередях, ожидающих обработки. Следить за тем, что rabbit справляется с очередью

**Ярлыки:**
- `queue` — имя очереди. Позволяет понять, какой именно consumer отстаёт.

### 3.4. Метрики сессий

#### Number of simultaneous sessions for MES API

**Зачем:** количество одновременных сессий операторов. Эта метрика в сочетании с latency дашборда позволяет отследить зависимость: «при N одновременных операторах дашборд начинает тормозить». Также полезна для расчета capacity.

## 4. План действий

### Этап 1. Инфраструктура мониторинга (неделя 1)

1. **Развернуть Prometheus** на отдельном EC2-инстансе. Prometheus будет центральным хранилищем метрик.
   - Настроить retention policy.

2. **Развернуть Grafana** на том же инстансе. Grafana — визуализация и дашборды.
   - Настроить авторизацию.
   - Подключить Prometheus.

3. **Развернуть Alertmanager** для маршрутизации алертов.
   - Настроить каналы: Telegram/Slack для команды, email для продакт-менеджера.
   - Настроить группировку алертов.

### Этап 2. Инструментирование API-сервисов (неделя 2)

4. **Shop API и CRM API (Java / Spring Boot):**
   - Подключить observability к API.

5. **MES API (C# / .NET):**
   - Подключить observability к API.

6. **Настроить Prometheus scraping targets** для всех трёх API.

### Этап 3. Мониторинг инфраструктуры (неделя 2, параллельно)

7. **EC2-инстансы (Shop API, CRM API, MES API):**
   - Установить `node_exporter` на каждый инстанс. Экспортирует CPU, память, диск, сеть.
   - Добавить targets в Prometheus.

8. **PostgreSQL (Shop DB, MES DB):**
   - Установить `postgres_exporter` рядом с каждой БД.
   - Добавить необходимые метрики в prometheus

9. **RabbitMQ:**
   - Настройка observability для брокера

10. **БД**
   - Настройка observability для БД

### Этап 4. Дашборды и алерты (неделя 3)

11. **Создать дашборды в Grafana:**
    - **System Overview:** общее здоровье — RED-метрики всех API на одном экране, CPU/память сервисов.
    - **RabbitMQ:** messages in flight по очередям, DLQ count, consumer count.
    - **Database Health:** connections, size, memory utilisation для обеих БД.

12. **Настроить алерты:**

    | Алерт | Условие | Серьёзность | Канал |
    |-------|---------|-------------|-------|
    | DLQ не пуста | `dlq_messages > 0` в течение 5 мин | Critical | Telegram + email |
    | Высокий error rate API | `5xx_rate > 5%` в течение 5 мин | Critical | Telegram |
    | Рост очереди RabbitMQ | `messages_in_flight > порог` и растёт 10 мин | Warning | Telegram |
    | CPU > 80% | `cpu_usage > 80%` в течение 10 мин | Warning | Telegram |
    | Память БД > 85% | `memory_usage > 85%` в течение 10 мин | Warning | Telegram |
    | Размер БД > 80% диска | `db_size / disk_total > 80%` | Warning | Email |
    | Latency MES дашборда > 3 сек | `p95 latency > 3s` для эндпоинта дашборда | Warning | Telegram |

### Этап 5. Distributed tracing (неделя 3–4, параллельно)

13. **Развернуть Jaeger**
    - Подключить трассировку в Shop API и CRM API через OpenTelemetry SDK для Java.
    - Подключить трассировку в MES API через OpenTelemetry SDK для .NET.
    - Инструментировать отправку/получение сообщений RabbitMQ

### Этап 6. Верификация и документация (неделя 4)

14. **Провести учения** — искусственно создать проблемные ситуации и проверить, что алерты срабатывают:
    - Остановить consumer RabbitMQ → проверить алерт на рост очереди.
    - Отправить невалидное сообщение → проверить алерт на DLQ.
    - Сгенерировать нагрузку на MES API → проверить алерт на CPU и latency.

15. **Написать документацию** для каждого алерта: что значит, какие шаги предпринять, к кому эскалировать.
