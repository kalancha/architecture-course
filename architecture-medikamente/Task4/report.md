# Задание 4. Оценка узких мест при миграции

## Ключевые процессы и компоненты миграции

| Процесс | Компоненты | Почему критично |
| --- | --- | --- |
| Запись пациента | Портал клиента, портал ресепшена, сервис записи, CRM | Это первый MVP-процесс и основной пользовательский сценарий |
| Медицинские данные | Медицинская карта, документы, лаборатория | Самые чувствительные данные; нужен строгий доступ и аудит |
| Оплата | ККМ, платёжный шлюз, 1С:Бухгалтерия | Ошибки влияют на деньги, налоги и доверие клиентов |
| Аналитика | Data Lake, BI, ML/AI/LLM | Нельзя передавать сырые PII и медицинские данные |
| Инфраструктура | Windows Server, будущий Kubernetes/сервисы, мониторинг | Пятикратный рост данных и филиалы требуют масштабируемости |

## Причины потенциальных узких мест

| Категория Исикавы | Возможные проблемы | Последствие |
| --- | --- | --- |
| Инфраструктура | Один локальный сервер, файловый режим 1С, общий диск, нет DR | Отказ сервера останавливает офис; рост филиалов перегружает текущий контур |
| Данные и хранилища | Excel/JPG/PDF без аудита, нет золотого источника, нет тегов, дублирование Excel и 1С | Данные расходятся, сложно мигрировать, невозможно доказать корректность доступа |
| Интеграции | ККМ через TCP/IP и OLE, будущий API лаборатории, нет API Gateway | Появляются хрупкие связи и риск передачи лишних данных партнёрам |
| Архитектура ПО | Смешение монолита, общих ресурсов и ручных процессов | Новые сервисы наследуют старую связанность и тормозят развитие |
| Безопасность и privacy | Нет RBAC/ABAC, шифрования, lineage, retention | Утечки, лишний доступ, невозможность расследовать инциденты |
| Люди и процессы | Маленький IT-отдел, CTO совмещает архитектуру, знания не документированы | Закон Конвея проявится в архитектуре: зоны ответственности будут размыты |
| Observability и тестирование | Нет аудита, алертов, нагрузочных и регрессионных проверок | Проблемы обнаружатся только после включения пользователей |

## Рекомендации и приоритеты

| Приоритет | Мера | Как снижает риск |
| --- | --- | --- |
| P0 | Сформировать HLD целевой архитектуры и домены данных | Даёт общее видение, границы сервисов и зоны ответственности |
| P0 | Провести инвентаризацию, тегирование и построить DFD/Data Lineage | Делает потоки данных управляемыми до начала миграции |
| P0 | Ввести IAM, RBAC/ABAC и аудит доступа до переноса медицинских данных | Снижает главный privacy-риск |
| P1 | Использовать Branch by Abstraction для записи, CRM и документов | Позволяет постепенно заменить Excel/общий диск, сохраняя старые процессы на переходе |
| P1 | Использовать Parallel Run для оплаты и интеграций с 1С/ККМ | Позволяет сверять старый и новый контур до полного переключения |
| P1 | Запустить мониторинг и алерты до пилотного запуска | Позволяет видеть ошибки, задержки и нетипичный доступ во время миграции |
| P1 | Подготовить план отката для MVP записи | Позволяет вернуть запись через ресепшен, если портал работает некорректно |
| P2 | Создать обезличенный аналитический слой | Даёт BI/ML/AI без нарушения минимизации данных |
| P2 | Документировать коллективные знания сотрудников | Снижает риск потери реальных бизнес-правил при автоматизации |

## Метрики для контроля миграции

| Метрика | Зачем нужна |
| --- | --- |
| Deployment Frequency | Проверить, не стала ли миграция тормозить поставку изменений |
| Lead Time for Changes | Найти задержки в разработке и согласовании изменений |
| MTTR | Оценить способность быстро устранять сбои |
| Ошибки записи на приём | Понять качество MVP-сценария |
| Доля ручных операций | Измерить фактическое снижение ручного труда |
| Количество отказов доступа по политикам | Проверить работу RBAC/ABAC и выявить неверные настройки |
| Количество обращений к тегам `PII` и `MEDICAL` | Видеть нагрузку на чувствительные данные и нетипичные действия |
| Время ответа портала и API | Контролировать производительность при росте клиентов |
| RPO/RTO для критичных данных | Проверить готовность к восстановлению |

## План перехода

1. Подготовка: инвентаризация данных, DFD, HLD, роли, теги, политики доступа.
2. MVP записи: портал клиента, портал ресепшена, сервис записи, CRM минимального профиля.
3. Branch by Abstraction: новые сервисы постепенно перехватывают запись и документы, старые Excel-процессы временно остаются как резерв.
4. Parallel Run: платежи и сверка с 1С/ККМ работают параллельно до подтверждения корректности.
5. Миграция медицинских документов: перенос сканов и карт в шифрованное хранилище с аудитом.
6. Интеграция лаборатории: запуск через API Gateway и mTLS с минимальными контрактами.
7. Аналитика: создание обезличенных витрин и запрет прямого доступа к сырым PII/MEDICAL.
8. Переключение: поэтапное отключение ручных Excel-процессов после сверки данных.
9. Откат: для каждого этапа фиксируется способ возврата на старый процесс и порядок повторной синхронизации.

## План отката для MVP записи

Если портал клиента или сервис записи работает некорректно:

1. Закрыть самостоятельную запись на портале и показать пациенту сообщение о записи через ресепшен.
2. Вернуть ресепшен к старому журналу записи на ограниченный период.
3. Экспортировать созданные в новом сервисе записи и сверить их с ручным журналом.
4. Устранить проблему, повторить регрессионные проверки и нагрузочный тест.
5. Снова открыть портал только после подтверждения корректности данных и уведомлений.
