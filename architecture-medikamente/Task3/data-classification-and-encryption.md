# Задание 3. Оценка Data Encryption at Rest and In Transit

## Классификация данных

| Данные | Теги | Состояние | Чувствительность | Профиль риска | В чём необходимость защиты | Инструменты защиты |
| --- | --- | --- | --- | --- | --- | --- |
| Ф. И. О., дата рождения, телефон, email пациента | `PII` | В покое, при передаче, при использовании | Высокая | Идентификация пациента, утечка персональных данных | Данные нужны для записи, договора и уведомлений, но не всем сотрудникам | TLS, шифрование БД/дисков, RBAC/ABAC, маскирование в UI, запрет PII в логах |
| Адрес, место работы/учёбы | `PII`, `RETENTION_LIMITED` | В покое, при передаче | Высокая | Избыточный сбор и бессрочное хранение | Часто не нужны для записи на приём; должны собираться только при наличии цели | Минимизация формы, отдельное согласие, срок хранения, шифрование, ограничение доступа |
| Хронические заболевания, анамнез, диагнозы | `MEDICAL`, `PII` | В покое, при передаче, при использовании | Критическая | Максимальный репутационный и регуляторный риск | Раскрытие медицинских сведений недопустимо; доступ только по назначению | Шифрование, ABAC по `doctor_id`/`appointment_id`, аудит чтения, минимальные API-ответы |
| Медицинская карта | `MEDICAL`, `PII` | В покое, при передаче, при использовании | Критическая | Несанкционированный просмотр, копирование, изменение | Медкарта является основным защищаемым активом | Шифрованная БД, контроль версий, аудит, резервные копии с шифрованием, запрет общего диска |
| Результаты анализов | `MEDICAL`, `PII` | При передаче, в покое | Критическая | Ошибочная выдача другому пациенту, утечка через API лаборатории | Интеграция с лабораторией должна передавать минимум данных | mTLS, API Gateway, object-level authorization, шифрованное хранилище, lineage |
| Договоры, согласия, акты | `CONTRACT`, `PII` | В покое, при передаче | Высокая | Содержат PII и факт обращения в клинику | Нужны пациенту, ресепшену и бухгалтерии, но не всем сотрудникам | Шифрованное объектное хранилище, RBAC, аудит скачивания, retention policy |
| Платежи, чеки, кассовые операции | `PAYMENT`, `PII` | В покое, при передаче | Высокая | Финансовые и персональные данные могут быть связаны с медицинской услугой | Нужны кассирам и бухгалтерии, но не медицинским специалистам | TLS, шифрование БД, интеграция с 1С через сервис, маскирование, аудит |
| Кадровые и зарплатные данные | `STAFF_PRIVATE` | В покое, при передаче | Высокая | Утечка персональных данных сотрудников | Доступ только бухгалтерии и руководству | Шифрование, отдельные роли, аудит, резервные копии |
| Складские данные и ТМЦ | `INTERNAL` | В покое, при передаче | Средняя | Нарушение операционной деятельности | Не медицинские данные, но важны для доступности бизнеса | RBAC, резервное копирование, TLS для интеграций |
| Сырые Excel/JPG/PDF из текущей системы | `PII`, `MEDICAL`, `CONTRACT`, `RETENTION_LIMITED` | В покое | Критическая | Общий диск, отсутствие аудита, копирование | Нужно мигрировать в управляемые хранилища и удалить лишние копии | Инвентаризация, классификация, миграция, шифрование, удаление дублей |
| Логи приложений и API | `INTERNAL`, возможные `PII` | В покое, при передаче | Средняя или высокая | Попадание PII/MEDICAL в логи | Логи нужны для эксплуатации, но не должны раскрывать данные | Обфускация идентификаторов, фильтры логирования, RBAC к Elastic, retention |
| BI-отчёты | `ANALYTICS_SAFE` | В покое, при использовании | Средняя | Повторная идентификация пациента | Аналитика должна работать без сырых PII/MEDICAL | Агрегация, обезличивание, маскирование, контроль малых выборок |
| ML/AI/LLM-наборы | `ANALYTICS_SAFE` | В покое, при использовании | Средняя или высокая | Обучение на чувствительных данных, утечка через модель | Нельзя использовать сырые данные пациента для обучения | Обезличивание, минимизация признаков, отдельная песочница, lineage датасетов |
| Резервные копии | Наследуют теги источника | В покое, при передаче | Такая же, как источник | Компрометация бэкапа раскрывает всю базу | Бэкапы часто содержат полный объём данных | Шифрование, отдельные ключи, ограничение доступа, проверка восстановления |

## Защита данных в покое

| Область | Мера |
| --- | --- |
| Базы данных Postgres/операционных сервисов | Шифрование дисков или шифрование уровня БД, роли БД по сервисам, запрет прямого пользовательского доступа |
| Файлы договоров, сканов и анализов | Перенос с общего диска в шифрованное объектное хранилище, аудит скачивания |
| 1С | Ограничение доступа к файловому режиму, резервное копирование, дальнейшая интеграция через сервисный слой |
| Резервные копии | Шифрование бэкапов, отдельные ключи, регулярная проверка восстановления |
| Аналитическое озеро | Хранение только `ANALYTICS_SAFE`, разделение зон raw/stage/mart, запрет сырых PII в витринах |
| Логи | Исключение PII/MEDICAL, обфускация идентификаторов, ограниченный срок хранения |

## Защита данных при передаче

| Поток | Мера |
| --- | --- |
| Пациентский портал → API Gateway | HTTPS/TLS, защита сессии, проверка пользователя |
| Портал ресепшена/врача/кассира → API Gateway | HTTPS/TLS, корпоративная аутентификация, MFA для чувствительных ролей |
| API Gateway → внутренние сервисы | TLS внутри контура, сервисная авторизация |
| Сервис лаборатории → внешняя лаборатория | mTLS, минимальные API-контракты, scopes, аудит каждого обмена |
| Сервис платежей → 1С/ККМ | Защищённый канал, сервисная учётная запись, журналирование операций |
| ETL → аналитический слой | Контролируемый поток, проверка тегов, передача только разрешённых полей |
| Бэкап → хранилище резервных копий | Шифрование канала и самого бэкапа |

## Защита данных при использовании

| Сценарий | Мера |
| --- | --- |
| Врач открывает медкарту | Проверка роли врача и связи с записью пациента; аудит чтения |
| Ресепшен смотрит запись | Отображать только данные, нужные для записи и договора |
| Кассир принимает оплату | Не показывать диагнозы и медицинскую историю |
| Бухгалтер строит отчёт | Использовать платёжные и договорные данные без медицинских деталей |
| Аналитик строит BI | Работать только с обезличенными наборами `ANALYTICS_SAFE` |
| IT расследует инцидент | Доступ к содержимому данных только через аварийную процедуру с аудитом |

| Средство | Тип | Назначение |

| IAM / Keycloak или аналог | ПО | Аутентификация, роли, группы, MFA, токены |
| Policy engine для ABAC | ПО | Проверка доступа по атрибутам `patient_id`, `doctor_id`, `appointment_id`, `data_tag`, `purpose` |
| API Gateway | ПО | Единая точка входа, scopes, лимиты, проверка контрактов |
| KMS / хранилище секретов | ПО/инфраструктура | Управление ключами шифрования и секретами сервисов |
| Шифрование дисков сервера | Аппаратно-программное | Защита БД, файлов, 1С и бэкапов в состоянии покоя |
| Шифрованное объектное хранилище | ПО/инфраструктура | Хранение договоров, сканов, анализов вместо общего диска |
| TLS/mTLS | Протоколы | Защита внутренних и внешних сетевых передач |
| Apache NiFi или аналог | ПО | Управляемые потоки данных, контроль передачи и происхождения |
| Apache Atlas / Collibra или аналог | ПО | Каталог данных, теги, lineage |
| Elastic | ПО | Централизованные логи и поиск событий |
| Victoria Metrics | ПО | Метрики, мониторинг, алертинг |
| Система резервного копирования | ПО/инфраструктура | Шифрованные бэкапы и проверка восстановления |
| CI/CD-проверки политик | ПО | Автоматический контроль, что API и сервисы корректно работают с тегированными данными |

## Автоматизированный контроль

| Контроль | Что проверяет | Реакция |
| --- | --- | --- |
| Проверка API-контрактов | API не возвращает поля с тегами `PII`, `MEDICAL`, `PAYMENT` без разрешённого scope | Блокировка релиза |
| Проверка политик RBAC/ABAC | Роль и атрибуты соответствуют доступу к данным | Ошибка теста или отказ доступа |
| Проверка логов | В логах нет Ф. И. О., телефона, email, диагнозов, результатов анализов | Ошибка релиза или алерт |
| Контроль выгрузок в аналитику | В Data Lake попадают только разрешённые поля и наборы `ANALYTICS_SAFE` | Остановка pipeline |
| Алерты нетипичного доступа | Массовое чтение карт, доступ не к своим пациентам, скачивание документов | Уведомление IT и ответственных за данные |
| Контроль сроков хранения | Временные и устаревшие данные удаляются или архивируются | Автоматическая задача lifecycle |

## Итоговая стратегия

1. Сначала провести инвентаризацию и тегирование данных.
2. Перенести чувствительные файлы с общего диска в управляемые хранилища.
3. Ввести шифрование для БД, файлов и бэкапов.
4. Ввести TLS/mTLS для внутренних и внешних потоков.
5. Ограничить доступ через RBAC/ABAC и теги.
6. Исключить PII/MEDICAL из логов и аналитики.
7. Для BI/ML/AI использовать только минимизированные и обезличенные данные.
8. Включить аудит, мониторинг и автоматические проверки политик в релизный процесс.
