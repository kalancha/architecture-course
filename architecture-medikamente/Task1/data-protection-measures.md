# Задание 1. Данные для защиты, теги и меры конфиденциальности

## Модель тегирования

| Тег | Назначение | Примеры данных | Базовая политика |
| --- | --- | --- | --- |
| `PUBLIC` | Данные без ограничений | Справочник услуг без ценовых условий пациента | Доступны всем внутренним пользователям и публичным сервисам, если нет других тегов |
| `INTERNAL` | Внутренние операционные данные | Складские заявки, расписание кабинетов | Доступ только сотрудникам по роли |
| `PII` | Персональные данные | Ф. И. О., дата рождения, телефон, email, адрес | Доступ по роли и цели обработки; запрет вывода в логи и аналитику без обезличивания |
| `MEDICAL` | Медицинские данные пациента | Диагнозы, хронические заболевания, назначения, заключения | Доступ только пациенту и назначенным медицинским специалистам; обязательный аудит |
| `PAYMENT` | Платёжные и кассовые данные | Сумма оплаты, чек, способ оплаты, связка с договором | Доступ кассирам и бухгалтерии; медицинским специалистам не отображается |
| `CONTRACT` | Договорные документы | Договор, согласия, акты | Доступ ресепшену, бухгалтерии и пациенту; аудит чтения/скачивания |
| `STAFF_PRIVATE` | Персональные данные сотрудников | Зарплата, кадровые документы, налоги | Доступ бухгалтерии и руководству |
| `ANALYTICS_SAFE` | Наборы, очищенные для BI/ML/AI | Агрегированные или обезличенные события | Доступ аналитикам; запрещена обратная идентификация пациента |
| `RETENTION_LIMITED` | Данные с ограниченным сроком хранения | Черновики форм, временные выгрузки, промежуточные файлы | Автоматическое удаление или архивирование по сроку |

## Роли и атрибуты доступа

| Субъект | Доступ разрешён | Доступ запрещён или ограничен |
| --- | --- | --- |
| Пациент | Только свои записи, договоры, анализы, заключения и уведомления | Данные других пациентов, внутренние комментарии, финансовая отчётность |
| Ресепшен | Записи на приём, контакты пациента, договорные данные в объёме процесса записи | Диагнозы, полный состав медицинской карты, зарплаты, склад |
| Медицинский специалист | Медицинские данные пациентов, с которыми есть назначение или лечение | Платёжные детали, данные пациентов вне назначения |
| Кассир | Платёжные операции, договор и минимальные идентификаторы пациента для оплаты | Медицинские заключения и история болезни |
| Бухгалтерия | Платежи, 1С, налоговые и кадровые данные | Медицинские карты и результаты анализов |
| Склад | ТМЦ и закупки | PII, медицинские и платёжные данные пациентов |
| Бизнес-аналитик | Только `ANALYTICS_SAFE` и агрегаты | Сырые `PII`, `MEDICAL`, `PAYMENT` без обезличивания |
| IT-администратор | Техническое сопровождение систем | Чтение бизнес-содержимого данных без аварийной процедуры и аудита |

Для MVP достаточно RBAC с отдельными атрибутами: `patient_id`, `appointment_id`, `doctor_id`, `department`, `purpose`, `data_tag`. В финальном состоянии политики стоит развивать до ABAC, чтобы доступ определялся не только ролью, но и контекстом.

## Способы защиты по классам данных

| Класс данных | Способ защиты | Где применять |
| --- | --- | --- |
| PII пациента | Шифрование, маскирование в интерфейсах, запрет логирования, токенизация идентификаторов | CRM, портал пациента, договоры, уведомления |
| Медицинские данные | Шифрование, строгий RBAC/ABAC, аудит чтения, минимизация выдачи через API | Медицинская карта, анализы, заключения |
| Платёжные данные | Шифрование, разделение доступа, маскирование в отчётах | Платёжный шлюз, 1С, кассовые операции |
| Договоры и сканы | Шифрование файлового/объектного хранилища, контроль скачивания, журналирование | Документооборот, архив договоров |
| Кадровые данные | Шифрование и отдельный контур доступа | 1С:Бухгалтерия и кадровый учёт |
| Аналитические наборы | Обезличивание, агрегация, маскирование, lineage до источника | BI, ML, AI, LLM |
| Логи | Обфускация идентификаторов, исключение PII/MEDICAL, ограничение доступа | Elastic, мониторинг, аудит |
| Резервные копии | Шифрование at rest, отдельные ключи, проверка восстановления | Базы данных, объектное хранилище, 1С |

## Инструменты и меры

| Область | Мера | Для чего нужна |
| --- | --- | --- |
| Управление потоками | Data Flow Diagram для каждого процесса | Показать, где данные создаются, передаются, хранятся и защищаются |
| Интеграция данных | Apache NiFi или аналогичный слой потоков | Управляемая передача данных, контроль происхождения, мониторинг потоков |
| Каталог и lineage | Apache Atlas, Collibra или аналог | Теги, владельцы данных, путь данных, связи между источниками и витринами |
| Доступ | IAM, RBAC/ABAC, политики на основе тегов | Ограничение доступа по роли, цели и контексту |
| API | API Gateway, проверка токенов, scopes, object-level authorization | Защита интеграции лаборатории и порталов от лишней выдачи данных |
| Хранение | Шифрование БД, дисков, объектного хранилища и бэкапов | Защита данных в покое |
| Передача | TLS для внутренних и внешних API, mTLS для лаборатории | Защита данных при передаче |
| Использование | Минимизация выборок, маскирование в UI, защищённые сессии | Снижение риска при обработке данных приложениями |
| Аналитика | Обезличенные слои данных и `ANALYTICS_SAFE`-наборы | BI/ML/AI без доступа к сырым персональным данным |
| Контроль релиза | Автотесты политик доступа и контрактов API | Проверка, что tagged data не попадают в запрещённые ответы |
| Аудит и мониторинг | События доступа, алерты по нетипичным действиям | Выявление инцидентов и расследование |

## Улучшенная модель обработки

1. Сырые данные пациента принимаются через портал или сотрудником ресепшена.
2. На входе применяется минимизация: обязательны только данные, нужные для записи и договора.
3. Данные классифицируются и получают теги `PII`, `MEDICAL`, `CONTRACT`, `PAYMENT` и другие.
4. Сервисы обращаются к данным через API и IAM, а не напрямую к общему диску.
5. Все обращения к чувствительным данным пишутся в аудит.
6. В аналитику уходят только обезличенные или агрегированные данные с тегом `ANALYTICS_SAFE`.
7. Для API лаборатории передаются только минимально необходимые категории данных, а результаты связываются с пациентом через контролируемый идентификатор.
8. Для временных файлов и устаревших данных действует срок хранения и автоматическое удаление.
