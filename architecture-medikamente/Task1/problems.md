# Задание 1. Анализ безопасности системы


## Конфиденциальные данные, которые не учтены как отдельные классы

| Класс данных | Примеры из кейса | Где сейчас появляются | Почему требуют защиты |
| --- | --- | --- | --- |
| Персональные данные пациента | Ф. И. О., дата рождения, телефон, email, адрес прописки | Excel-журналы, договоры, сканы, карточки пациентов, Outlook | Позволяют идентифицировать пациента; относятся к PII и персональным данным |
| Медицинские данные | Хронические заболевания, медицинская карта, заключения, результаты анализов | Excel, JPG/PDF-сканы, общий диск, рабочие файлы специалистов | Являются наиболее чувствительной категорией; раскрытие приводит к максимальному репутационному и регуляторному риску |
| Данные договоров | Договор с пациентом, состав услуг, даты посещений | Сканы, общий диск, файлы ресепшена | Связывают пациента с фактом обращения за медицинской услугой |
| Платёжные данные | Платежи, кассовые операции, связка пациента с оплатой | ККМ, 1С:Бухгалтерия, Excel-дубли | Нужны для бухгалтерии, но не должны быть доступны медицинским специалистам и всем пользователям общего диска |
| Кадровые и зарплатные данные | Данные сотрудников, зарплаты, налоги | 1С:Бухгалтерия | Требуют отдельной зоны доступа бухгалтерии и руководства |
| Коммерческие и операционные данные | ТМЦ, закупки оборудования, складские остатки | 1С:Торговля и склад | Не являются медицинскими данными, но важны для целостности и доступности бизнеса |
| Аналитические наборы | Данные для BI, ML, AI, LLM | Сейчас формируются вручную из Excel | Нельзя передавать в аналитику сырые PII/медицинские данные без минимизации и обезличивания |

## Проблемные зоны As-Is

| N | Проблема | Где проявляется | Риск | Практика из курса |
| --- | --- | --- | --- | --- |
| 1 | Нет инвентаризации и классификации данных | Excel, сканы, 1С, общий диск | Компания не знает, какие чувствительные данные где хранятся и кто ими пользуется | Privacy by Design, тегирование |
| 2 | Данные лежат рядом с пользователями | Общий диск, локальные файлы, JPG/PDF | Высокий риск несанкционированного копирования, удаления и утечки | Privacy by Default, контроль доступа |
| 3 | Нет аудита чтения и изменения данных | Файлы и текущие IT-системы | Нельзя расследовать инциденты и выявлять нетипичные действия | Data Lineage, мониторинг, аудит |
| 4 | Доступ ограничен только доменной аутентификацией | Active Directory без доменной модели доступа к данным | Любой пользователь с доступом к файловому ресурсу может увидеть лишние данные | RBAC/ABAC, теги данных |
| 5 | Нет управляемых потоков данных | Внутренние передачи между Excel, 1С, Outlook, сканами | Потоки нельзя проверить, защитить и валидировать при изменениях | Data Flow Management, DFD |
| 6 | Сбор данных избыточен | Расширенная форма пациента: адрес, работа/учёба, хронические заболевания | Нарушается принцип минимизации; лишние данные увеличивают область атаки | Data Minimization |
| 7 | Данные дублируются вручную | Кассиры ведут Excel и 1С | Ошибки, рассинхронизация, недостоверный золотой источник | Слои данных, золотой источник |
| 8 | Нет разделения слоёв данных | Сырые сканы, рабочие таблицы и отчёты лежат рядом | Сырые чувствительные данные попадают в аналитику и отчёты без обработки | Сырые/обработанные данные/золотой источник |
| 9 | Нет защиты данных в разных состояниях | Файлы на диске, передача по сети, работа в памяти приложений | Нет единой стратегии шифрования at rest/in transit/in use | Защита данных в покое, при передаче и использовании |
| 10 | Нет безопасной модели интеграции с лабораторией | Планируемое API лаборатории | Партнёр может получить лишние категории данных или данные другого пациента | API-контракты, минимизация, ABAC |
| 11 | Нет безопасной аналитической платформы | Будущие BI, ML, AI, LLM | Риск обучения моделей на сырых персональных и медицинских данных | Обезличивание, маскирование, Data Lineage |
| 12 | Нет сроков хранения и удаления | Excel, сканы, 1С, архивы | Данные накапливаются бессрочно и остаются доступными после утраты цели обработки | Data Minimization, lifecycle |
| 13 | Нет мониторинга доступа к чувствительным данным | Текущие файлы и будущие сервисы | Компания не узнает о подозрительных действиях до инцидента | Аудит, алертинг, tagged data monitoring |
| 14 | Единый локальный сервер является точкой отказа | Windows Server 2022 в офисе | Отказ сервера или инцидент в офисе остановит ключевые процессы | Оценка узких мест, доступность |

## Приоритеты устранения

| Приоритет | Что сделать | Почему сначала это |
| --- | --- | --- |
| P0 | Провести инвентаризацию данных, классификацию и тегирование | Без понимания классов данных нельзя корректно настроить доступы, шифрование, аудит и минимизацию |
| P0 | Убрать общий неаудируемый доступ к медицинским картам | Это главный риск утечки наиболее чувствительных данных |
| P0 | Ввести RBAC/ABAC для пациентов, ресепшена, врачей, кассиров, бухгалтерии и IT | Доступ должен определяться ролью, назначением и атрибутами, а не фактом входа в домен |
| P1 | Построить управляемые DFD и Data Lineage для основных процессов | Потоки данных нужны для API лаборатории, MVP портала и будущих интеграций |
| P1 | Ввести шифрование at rest и in transit | Защищает данные на дисках, в базах, резервных копиях и сетевых передачах |
| P1 | Создать золотые источники для пациентов, записей, медицинских документов и платежей | Снижает дублирование Excel/1С и ошибки ручного учёта |
| P2 | Подготовить обезличенные/маскированные наборы для BI, ML, AI | Аналитика должна работать без доступа к сырым PII и медицинским данным |
| P2 | Настроить мониторинг, аудит и алерты по тегированным данным | Позволяет выявлять нетипичные действия и подтверждать соблюдение политик |
